# F-003: Streamlined Local Startup and Chromebook Validation

## Objective

Make the F-002 local connectivity workflow easier to run, especially on the intended Chromebook host, now that the Chromebook-to-iPad path has been validated manually.

## User Value

The GM should not need to remember several setup commands or manually discover network details every time TabletopFog is started. This follow-up should reduce setup friction before map or fog features are added.

## Non-Goals

- Do not add map loading.
- Do not add fog rendering.
- Do not add save/load.
- Do not add auth, cloud hosting, pairing codes, tokens, dice, sheets, or combat automation.
- Do not replace F-007 quality-of-life work such as QR codes or fullscreen player polish unless explicitly approved later.

## Acceptance Notes

- A single command starts the local workflow or clearly prints the remaining manual step.
- The command detects and prints the current LAN IP when possible.
- The command regenerates or verifies the development HTTPS certificate for the current LAN IP.
- The command starts the HTTPS server on `0.0.0.0:3000`.
- Startup output clearly prints the GM URL and player URL.
- Startup output identifies the generated certificate file that must be installed or trusted on iPhone/iPad.
- The flow works on MacBook Pro.
- The flow preserves the validated Chromebook Linux development environment path.
- iPad can connect to the Chromebook host using the Chromebook LAN IP after following the printed setup steps.
- Any Chromebook, firewall, OpenSSL, certificate trust, or LAN isolation issues are documented.

## Clarification Notes

- F-002 quick MacBook test worked.
- Chromebook-to-iPad validation was completed during F-002 with ChromeOS Linux port forwarding for TCP port `3000`.
- Chromebook-local access should use `https://localhost:3000/...` or `https://penguin.linux.test:3000/...`; separate player devices should use the Chromebook Wi-Fi LAN IP.
- Current manual startup steps are too many:
  - discover LAN IP,
  - run `npm run cert -- --ip=<LAN-IP>`,
  - run `npm run dev`,
  - copy or type the player URL.
- Preferred next improvement is a command such as `npm run local` that detects the LAN IP, generates the cert, starts the server, and prints useful URLs.
- The certificate trust flow should not be over-polished before Chromebook behavior is tested once.
- If automatic IP detection reports the wrong address, especially from Chromebook Linux, use `npm run local -- --ip=<LAN-IP>` with the host's Wi-Fi LAN IP.
- The local startup command should reuse a still-valid certificate so iPad trust does not need to be repeated on every server start.

## SDLC Agent Outputs

### Architecture Review

- Add `npm run local` as a Node startup wrapper instead of duplicating server logic.
- Reuse the existing HTTPS Express/Socket.IO server through `createTabletopFogServer()`.
- Keep `npm run dev` available as the raw server command.
- Reuse the certificate helper logic, but make it importable so startup orchestration can detect IPs and ensure the certificate before listening.
- Preserve trusted-LAN assumptions and avoid auth, QR codes, pairing, map, fog, or player UI changes.

### Implementation Design

- `scripts/create-dev-cert.js` exports certificate and LAN IP helpers while preserving the existing `npm run cert -- --ip=<LAN-IP>` CLI.
- `scripts/start-local.js` parses optional `--ip=` overrides, detects LAN IP candidates, reuses the current certificate when it is still valid for those IPs, regenerates only when needed, starts the existing server on `0.0.0.0:3000`, and prints startup details.
- `package.json` exposes the wrapper as `npm run local`.
- The command prints all detected player URL candidates instead of choosing one silently.

### Test and Validation Design

- Automated tests cover startup detail generation, output formatting, fallback output when no LAN IP is detected, explicit `--ip=` parsing, loopback/deduplication filtering, certificate missing-file regeneration decisions, and the non-hanging startup orchestration path.
- Manual validation must include MacBook Pro local command startup, iPhone/iPad player URL checks, and Chromebook host to iPad player view.
- Long-running `npm run local` is validated manually rather than inside `node --test`.

### UX and Workflow Review

- Startup output is short and actionable: listening URL, GM URL, player URL candidates, certificate path, certificate status, and Chromebook port-forwarding/local URL caveats.
- The command supports explicit IP override for Chromebook Linux cases where detected container networking does not match the Wi-Fi LAN IP.
- QR codes and broader sharing UI remain out of scope for F-003.
