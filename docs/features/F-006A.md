# F-006A: GM Map Zoom and Grid Calibration

## Objective

Add the first Encounter Workspace preparation tools needed before manual fog:
GM-local map zoom controls and a 5 ft by 5 ft battlemap grid overlay that can
be positioned, locked, and unlocked.

## User Value

The GM can line up TabletopFog's workspace view with the map's intended battle
grid before drawing fog. This gives the GM a stable reference for future hide
and reveal work without adding full VTT combat features.

## Non-Goals

- Do not add fog drawing, hiding, revealing, brushes, or shape tools yet.
- Do not add tokens, movement, snapping, measurement tools, range tools, or
  rules automation.
- Do not add dynamic lighting, vision cones, or character-specific visibility.
- Do not change Player Display behavior unless explicitly approved during
  F-006A clarification.
- Do not persist grid or zoom state unless explicitly approved during F-006A
  clarification; persistence remains part of F-007 by default.
- Do not change storage, API routes, or shown-to-players terminology unless
  implementation review finds a narrow compatibility need.

## Acceptance Notes

- Grid and zoom controls live in the Encounter Workspace.
- The map remains the dominant workspace element.
- GM can zoom the selected encounter map in and out.
- GM map zoom is local to the GM browser view and does not change the Player
  Display.
- A square 5 ft grid overlay can be shown over the selected encounter map.
- The grid remains fixed in screen/workspace space while the map zooms in and
  out beneath it.
- GM can adjust the grid placement until it lines up with the battlemap.
- GM can lock the grid after alignment.
- When locked, accidental grid movement is prevented.
- GM can unlock the grid and adjust it again.
- Opening or editing the grid on an encounter does not change the Player
  Display.
- `Show to Players` remains the only action that changes what players see.
- The controls remain usable on desktop, Chromebook-sized, and narrow viewports.
- No fog, tokens, notes, initiative, dynamic lighting, or VTT-style tools are
  introduced.

## Affected Systems

- GM View: Encounter Workspace controls, map viewport behavior, and grid overlay
  interaction.
- Player Display: no intended changes unless clarified otherwise.
- Local server: no intended changes unless grid state is shared or persisted.
- Map/fog rendering: map zoom and grid overlay composition in the workspace
  renderer; no fog mask yet.
- Storage: no intended changes unless grid state persistence is explicitly
  approved.
- Documentation: feature tracker, acceptance tests, and future F-006 planning.

## Validation Plan

Open an Encounter Workspace with a selected encounter map. Confirm zoom in and
zoom out affect only the GM workspace map. Show a 5 ft grid overlay, adjust its
placement, zoom the map, and confirm the grid remains fixed while the map zooms
beneath it. Lock the grid and confirm accidental grid movement is blocked.
Unlock the grid and confirm it can be adjusted again. Confirm none of these
actions change the Player Display or shown-to-players encounter.

Repeat the flow at desktop, Chromebook-sized, and narrow viewport widths.

## Open Questions

- Should the grid be GM-only for F-006A, or should there eventually be a
  separate explicit option to show it on the Player Display?
- What grid placement controls should the first version support: drag only,
  nudge buttons, keyboard arrows, numeric offsets, cell-size adjustment, or a
  minimal subset?
- Does "5 ft by 5 ft" mean a fixed labeled scale only, or should cell pixel size
  be adjustable per encounter to match maps with different resolutions?
- Should grid lock state reset when leaving the Encounter Workspace, or remain
  in memory while the campaign is open?
- Should F-006A persist grid calibration immediately, or defer persistence to
  F-007 with fog state?
- Should grid state belong to the selected encounter even before persistence,
  or should it be temporary GM viewport state until fog uses it?
- Should GM map zoom reuse the existing player viewport controls exactly, or
  use a smaller Encounter Workspace-specific control set?
