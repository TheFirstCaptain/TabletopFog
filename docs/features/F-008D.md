# F-008D: Expanded Fog Shape Tools

## Objective

Explore and define fog tools beyond rectangles, starting with circular hide and
reveal areas whose size can be selected quickly or entered precisely.

## User Value

The GM can hide or reveal round rooms, spell areas, cave sections, and irregular
table moments with less manual rectangle stacking while keeping fog editing
fast during play.

## Non-Goals

- Do not add tokens, vision cones, dynamic lighting, line of sight, or
  automation.
- Do not add a full drawing program or arbitrary shape editing.
- Do not add player controls.
- Do not change which encounter is `Shown to Players`.
- Do not replace the existing rectangle Hide and Reveal tools.
- Do not add cloud storage, accounts, or shared online play.
- Do not migrate storage/API terminology such as `maps`, `activeMapId`, or
  `fog`.

## Acceptance Notes

- The first discussion target is circular fog tools with adjustable size.
- Circle size can be adjusted with a slider and typed directly into an input.
- Circle size controls must support fine-grained values rather than limiting
  the GM to coarse 25% increments.
- Any preset sizes, if added, are shortcuts and do not replace direct entry or
  slider control.
- Any new fog shape must support both hide and reveal behavior, or the feature
  must explicitly document why it starts with only one direction.
- New fog tools live in the Encounter Workspace Fog tools group.
- Shape tools must respect the selected/editing versus `Shown to Players`
  distinction.
- If the selected encounter is currently `Shown to Players`, completed fog
  changes update the Player Display live.
- If the selected encounter is not `Shown to Players`, completed fog changes do
  not change the Player Display.
- New fog operations autosave to local campaign storage.
- Existing rectangle operations, undo behavior, and Clear Fog behavior must
  remain compatible.
- Player Display remains read-only.
- The UI must stay calm and usable for a standing GM; avoid dense toolbars or
  precision-only controls.

## Clarification Notes

- Scaffolded on 2026-07-04 after the user noted that older circular fog ideas
  had been deferred and that more complicated fog tools may be needed.
- The user specifically mentioned circles with different sizes as the starting
  point for discussion.
- The user clarified on 2026-07-04 that circle sizing should support typed
  entry and slider adjustment, and should not be limited to 25% increments.
- F-006 intentionally started with simple rectangle geometry before more
  advanced fog tools. F-008D is the proposed place to revisit circles, brush-like
  behavior, and other non-rectangle shapes.

## Harness Compliance

Harness Version: 1

## SDLC Tier

Tier: 3.

Rationale: Expanded fog tools are expected to affect GM fog input, persisted fog
operation shapes, filesystem writes, server state, Socket.IO synchronization,
Player Display projection, player read-only guarantees, and map/fog rendering
semantics. Those areas are Tier 3 under
`docs/decisions/decision-004-mandatory-subagent-sdlc.md`.

## Affected Systems

- GM view: may add circle or brush controls in the Encounter Workspace Fog tools
  group.
- Player view: receives rendered fog changes only for the encounter currently
  `Shown to Players`.
- Local server: may accept new validated fog operation types.
- Map/fog rendering: must render new fog shapes consistently for GM and Player
  Display.
- Storage: must preserve and recover any new persisted fog operation shapes.
- Documentation: roadmap, tracker, acceptance tests, README current status, and
  storage-format notes after implementation.

## Validation Plan

Define coverage after clarification. Expected validation should include shape
geometry, slider and typed size entry, fine-grained size values, hide and reveal
semantics, operation ordering with existing rectangles, undo compatibility,
Clear Fog compatibility, selected unshown isolation, shown-player live updates,
autosave persistence, malformed shape recovery, Player Display read-only
behavior, and responsive GM controls.

Run focused Node and Chromium tests for the chosen tool design, then run
`npm run harness:check`, `npm run test:browser`, and `npm run quality` before
marking the feature complete.

## Open Questions

- Should F-008D start with circles only, or include a brush-like tool?
- Resolved: circle size should support both slider adjustment and typed direct
  entry.
- Resolved: circle size should not be limited to 25% increments.
- What minimum, maximum, default, and step values are useful for V1?
- Should circle drawing be click-to-place, drag-to-size, or center-and-radius
  drag?
- Should circle tools have separate `Hide circle` and `Reveal circle` buttons,
  or share shape selection with existing Hide and Reveal modes?
- How should circle operations be represented in persisted fog state while
  preserving forward compatibility?
- Should F-008D wait until F-008C undo is implemented so new shape operations
  are undoable from the start?
