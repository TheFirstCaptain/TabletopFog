# F-006F: Fog Tool Polish and Validation

## Objective

Finish the manual-fog workflow by tightening UX, edge cases, responsive
behavior, and completion evidence for the F-006 umbrella feature.

## User Value

The GM gets a coherent, reliable fog workflow that is usable at the table rather
than a set of isolated drawing primitives.

## Non-Goals

- Do not add new fog shapes beyond the approved F-006 scope.
- Do not add brushes, undo, operation history, or saved fog presets.
- Do not persist fog to campaign storage; that remains F-007.
- Do not migrate storage or API terminology unless separately approved.
- Do not add dynamic lighting, tokens, vision cones, notes, or initiative.

## Acceptance Notes

- Fog controls remain compact and subordinate to the map.
- Hide, Reveal, and Clear Fog states are understandable in the Encounter
  Workspace.
- When the GM zooms the workspace map, the GM can pan the map locally to reach
  off-screen map areas.
- GM workspace panning is local to the GM browser tab and never changes the
  Player Display, campaign state, fog state, grid state, or persistence.
- GM workspace panning arbitrates cleanly with fog drawing and grid dragging.
- Pointer, keyboard, cancellation, failed-image, and small-drag behavior are
  reviewed and documented.
- Fog tools remain usable on desktop, Chromebook-sized, and narrow viewports.
- Player Display remains read-only and receives only fog for the shown
  encounter.
- Editing fog on an unshown selected encounter never changes the Player Display.
- Editing fog on the shown encounter updates the Player Display live.
- Fog remains in-memory only.
- F-006 umbrella acceptance notes, tracker state, README, roadmap, and
  acceptance tests are reviewed for consistency.

1. Audit Current Fog Workflow

   Review the completed F-006A through F-006E behavior as one workflow:
   GM-local zoom and pan, grid alignment, Hide rectangle, Reveal rectangle,
   Clear Fog, shown-to-players sync, unshown prep isolation, failed-image
   behavior, and read-only Player Display.

2. Validate Interaction Edges

   Confirm or document the current behavior for:

   - Hide and Reveal active-state toggling.
   - Escape cancellation.
   - Pointer cancellation.
   - Tiny drags smaller than the 6 px threshold.
   - Off-map drag starts.
   - GM workspace map panning while zoomed.
   - Grid input arbitration while fog drawing is active.
   - Pan input arbitration while fog drawing or unlocked grid dragging is
     active.
   - Clear Fog confirmation cancel and accept.
   - Clear Fog after image load failure.
   - Browser reload/reconnect behavior where current automation already covers
     the surrounding state.

3. Review Keyboard Accessibility Boundary

   F-006C and F-006D intentionally shipped pointer-drawn rectangles. F-006F
   should explicitly review keyboard behavior and either:

   - document that keyboard rectangle drawing is deferred to F-008 or a future
     accessibility-focused feature, or
   - add a small reviewed improvement if a simple, table-appropriate interaction
     is available without expanding scope.

4. Responsive and Layout Review

   Re-check desktop, Chromebook-sized, and narrow workspace layouts after all
   fog controls and GM-local panning are present. The map should remain
   dominant, panning should not introduce clipping or overlap, and the Fog tools
   should remain compact.

5. Documentation and Umbrella Completion

   Sweep F-006, F-006F, the feature tracker, roadmap, acceptance tests, README,
   and material findings. F-006 should move to completion only after the full
   manual-fog workflow is validated and F-006F is complete.
