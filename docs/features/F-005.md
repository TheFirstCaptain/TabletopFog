# F-005: Static Active Map Display Polish

## Objective

Harden and polish the selected active campaign map display on both GM and player views after F-004's minimal display path exists.

## User Value

The GM and players get a more reliable table display before fog-of-war controls are added.

## Non-Goals

- Do not add fog controls.
- Do not add new campaign library behavior beyond F-004.
- Do not add tokens.
- Do not add notes.
- Do not add initiative tracking.

## Acceptance Notes

- GM and player views use consistent active-map rendering behavior.
- Map scales reasonably to the available screen across desktop, iPad, and TV-like landscape viewports.
- Player view can zoom its active map in and out without changing the GM view's viewport.
- GM and player viewport state remains local to each view so later view-specific behavior can diverge cleanly.
- Active-map display handles refresh, reconnect, and image-load error states cleanly.
- Player receives only the active-map display state needed for read-only rendering.
- Player view remains read-only.
- No fog controls are exposed yet.

## Clarification Notes

- Start F-005 while F-004's remaining physical-device validation is pending; the physical checks can validate the combined display path.
- Use a shared rendering foundation for GM and player views, but allow each view to have distinct surrounding UI and behavior.
- Use canvas rendering to establish the map-rendering path needed by later fog work.
- Default initial fit is `contain`, preserving the complete map with letterboxing when aspect ratios differ.
- Do not add selectable fit modes in F-005.
- Player zoom is local display state. Zooming the player view must not change GM viewport state or shared campaign/session state.
- Preserve room for unique GM and player viewport requirements rather than treating their complete view state as synchronized.
- Validate desktop, iPad-like, and TV-like landscape viewports in automation where practical; physical iPad and Chromebook validation remains a manual boundary.
