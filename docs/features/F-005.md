# F-005: Save and Load State

## Objective

Persist the current map and fog state locally and reload it later.

## User Value

The GM can pause and resume a session without recreating map and fog state.

## Non-Goals

- Do not add cloud storage.
- Do not add user accounts.
- Do not add campaign management.
- Do not add multiple-scene workflows unless separately planned.

## Acceptance Notes

- Save current map and fog state locally.
- Reload a previous session.
- Restored player view matches the saved revealed state.
- Storage format is documented well enough to migrate later if needed.

## Affected Systems

- GM view: save/load controls.
- Player view: displays restored state.
- Local server: loads and serves restored state.
- Map/fog rendering: restores map and reveal state.
- Storage: local files or lightweight local storage.
- Documentation: update acceptance tests and decisions if storage architecture is significant.

## Validation Plan

Create a map/fog state, save it, restart or reload the app, load the saved state, and verify GM/player views match expected state.

## Open Questions

- Should the first persistence target be a local JSON file, browser storage, or a server-side data file?
- How should map image data be referenced or bundled with saved state?
