# F-009B: AI Harness Compliance and Review Evidence

## Objective

Make mandatory AI-development stages, review findings, dispositions, and final
validation evidence structured and mechanically verifiable.

## User Value

Maintainers can distinguish completed engineering review from documentation that
only claims a workflow occurred.

## Non-Goals

- Do not replace technical judgment with a checklist.
- Do not require the same domain checks for every unrelated change.
- Do not change product behavior.
- Do not implement continuous integration; F-009C owns remote enforcement.

## Acceptance Notes

- Feature documents contain every mandatory SDLC stage as a non-empty structured
  section.
- Material findings record reviewer or stage, severity, recommendation,
  disposition, validation evidence, and residual risk.
- The main agent records all material findings, including rejected or deferred
  recommendations.
- Stage-specific checklists cover persistence/data integrity, external input,
  security/trust boundaries, error handling/observability, module boundaries,
  dependencies, browser behavior, test realism, and resource cleanup when
  relevant.
- Definition of Done has hard gates for required quality evidence.
- A final-diff review occurs after implementation and test changes are complete.
- A checker detects missing stages, empty evidence, inconsistent tracker status,
  and unresolved blocking findings.
- The checker is included in the F-009A local quality command.

## Feature Readiness Checklist

- Clarification complete.
- Acceptance notes drafted.

## Clarification Notes

- Preserve concise feature records while making material evidence explicit.
- Use a versioned, exact Markdown contract in each governed feature document;
  do not add duplicative evidence sidecars.
- Enforce evidence progressively by tracker phase: clarification before planning,
  design reviews before building, coding evidence before validation, and final
  reviews plus validation before Done.
- Critical and high findings block completion until fixed. Medium findings block
  until fixed or explicitly accepted/deferred by the user with a linked bug,
  engineering item, or decision plus an ISO review/expiry date. Low findings
  require a recorded disposition but do not otherwise block.
- Require every material finding to retain its stage, reviewer identity,
  severity, disposition, evidence, and residual risk.
- If required independent review or subagents are unavailable, move the feature
  to Blocked. A human may replace a subagent; the main agent cannot silently
  waive its own independent review. Any exception is user-approved and limited
  to the current feature.
- Grandfather F-001 through F-004 plus the F-009 umbrella and F-009A in a strict
  legacy baseline. Proposed features are not grandfathered and must adopt
  Harness Version 1 before leaving Proposed.
- Use exact headings and a simple findings table that can be checked without a
  general-purpose Markdown parser.
- Add `harness:check` to `npm run quality` before coverage.
- Update Decision 004 only when the new evidence model is ready to become active
  policy.
