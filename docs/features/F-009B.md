# F-009B: AI Harness Compliance and Review Evidence

## Objective

Make mandatory AI-development stages, review findings, dispositions, and final
validation evidence structured and mechanically verifiable.

## User Value

Maintainers can distinguish completed engineering review from documentation that
only claims a workflow occurred.

## Non-Goals

- Do not replace technical judgment with a checklist.
- Do not require the same domain checks for every unrelated change.
- Do not change product behavior.
- Do not implement continuous integration; F-009C owns remote enforcement.

## Acceptance Notes

- Feature documents contain every mandatory SDLC stage as a non-empty structured
  section.
- Material findings record reviewer or stage, severity, recommendation,
  disposition, validation evidence, and residual risk.
- The main agent records all material findings, including rejected or deferred
  recommendations.
- Stage-specific checklists cover persistence/data integrity, external input,
  security/trust boundaries, error handling/observability, module boundaries,
  dependencies, browser behavior, test realism, and resource cleanup when
  relevant.
- Definition of Done has hard gates for required quality evidence.
- A final-diff review occurs after implementation and test changes are complete.
- A checker detects missing stages, empty evidence, inconsistent tracker status,
  and unresolved blocking findings.
- The checker is included in the F-009A local quality command.
