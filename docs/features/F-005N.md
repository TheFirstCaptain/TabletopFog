# F-005N: Encounter Workspace Map Density

## Objective

Reduce non-map chrome and whitespace in the Encounter Workspace so the selected
map appears earlier and occupies more of the first viewport before F-006 fog
tools are added.

## User Value

The GM can focus on the encounter map immediately after opening an encounter,
with navigation, status, and running actions still available but visually
subordinate to the map.

## Non-Goals

- Do not add fog controls.
- Do not add new workspace workflows.
- Do not change canvas rendering behavior except for layout sizing if required.
- Do not change storage, API routes, or campaign metadata.
- Do not change the selected/editing versus shown-to-players model.
- Do not change Campaign Library or encounter gallery behavior.
- Do not add animations or transitions.
- Do not expand TabletopFog toward full VTT scope.

## Acceptance Notes

- Encounter Workspace map starts higher in the first viewport on desktop and
  Chromebook-sized screens.
- Non-map workspace chrome uses less vertical space when an encounter is open.
- `Back to Campaign` remains obvious and reachable but is visually subordinate
  to the selected map.
- The selected encounter title, selected/shown status, and running action remain
  readable and accessible.
- The `Shown to Players` clear action remains explicit and visually actionable.
- The map stage occupies more of the visible workspace without overlap,
  clipping, or horizontal overflow.
- The `Future Tools` placeholder is further reduced or repositioned as a quiet
  reserved affordance until F-006 replaces it with real tools.
- Narrow viewport behavior remains usable and does not hide required navigation
  or running actions.
- Opening or editing an encounter still does not change the Player Display.
- No fog controls, storage changes, API changes, workflow changes, animations,
  or transitions are introduced.

## Clarification Notes

- This feature exists because the current Encounter Workspace leaves too much
  empty space above and around the map.
- Prefer making the top workspace area compact rather than adding new controls.
- Consider placing the running action closer to the selected/shown status or
  otherwise avoiding a full-width action row.
- Keep the app-level page header out of scope unless review shows it is the
  main cause of map displacement.
- Keep the map as the dominant first-viewport element.
- 2026-06-30: User clarified that the Encounter Workspace header should be very
  small, the map should dominate the page, and a fixed quiet tool area is
  preferable to floating controls. Keep the broader app/page header unchanged
  unless it is clearly the main source of map displacement.
