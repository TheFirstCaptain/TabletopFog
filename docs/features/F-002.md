# F-002: Local Connectivity Spike

## Objective

Prove that the GM machine can host TabletopFog locally and an iPad can connect over the same Wi-Fi network.

## User Value

This validates the core MVP assumption before any map, fog, or storage complexity is added.

## Non-Goals

- Do not add map loading.
- Do not add fog rendering.
- Do not add save/load.
- Do not add auth, cloud hosting, tokens, dice, sheets, or combat automation.

## Acceptance Notes

- GM page is served from a local server.
- Player page is served from the same local server.
- Both pages are served over HTTPS.
- iPhone and iPad can connect to the MacBook Pro host using its LAN IP address during primary early validation.
- iPad can connect to the Chromebook host using its LAN IP address during intended-use validation.
- GM can trigger one simple state change.
- Player view updates live without manual refresh.
- The same workflow is tested on home Wi-Fi and a friend's Wi-Fi.

## Clarification Notes

- Primary development host: MacBook Pro.
- Primary early client test devices: iPhone and iPad.
- Intended table host: Chromebook.
- Intended table client: iPad.
- F-002 should validate the MacBook Pro host path first, then repeat the core flow on the Chromebook before the feature is considered complete.
- MacBook Pro certificate workflow uses the project `npm run cert -- --ip=<LAN-IP>` helper for the first implementation spike.
- The F-002 synced state change should be neutral, such as a counter, status, or color change. It should not be fog-themed and should not imply map or fog behavior.
- Chromebook validation may assume Linux development environment is enabled with Node.js and npm available.

## Architecture Review

F-002 uses:

- `server/index.js` for HTTPS Express routes and Socket.IO sync.
- `server/state.js` for a tiny in-memory counter state.
- `server/certs.js` for local certificate loading.
- `public/gm.html` and `public/gm.js` for the GM counter controls.
- `public/player.html` and `public/player.js` for the read-only player display.
- `scripts/create-dev-cert.js` for development-only self-signed certificates.
- `test/` for Node tests covering HTTPS routes and live sync.

The server binds to `0.0.0.0` for LAN clients. State is in memory only. No map, fog, persistence, auth, cloud, token, dice, sheet, or combat behavior is included.

## UX Review

The synced state is a neutral shared counter. `/gm` displays connection status, current counter, and one increment button. `/player` displays connection status and the counter only. The player view exposes no controls.
