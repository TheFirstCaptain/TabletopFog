# F-007C: Persistence Validation and Recovery States

## Objective

Harden campaign reload behavior for malformed persisted fog, missing map assets,
and storage-format documentation.

## User Value

The GM gets safe, understandable recovery behavior when local campaign files are
edited, damaged, or moved outside the app.

## Non-Goals

- Do not add cloud repair or sync.
- Do not add accounts.
- Do not add map replacement.
- Do not add multiple fog states.
- Do not hide local data problems silently.

## Acceptance Notes

- Malformed persisted fog is isolated to the affected encounter where practical
  and reported without destructive rewrites.
- Missing map assets are isolated to the affected encounter where practical and
  reported clearly when reopening a campaign.
- Valid campaigns and valid encounters remain usable when unrelated local files
  are invalid, where practical.
- Storage format and migration expectations are documented.
- Recovery behavior preserves unknown forward-compatible metadata.
- Campaign library listing must keep valid campaigns visible when another
  campaign or encounter has recovery diagnostics.
- Player Display must receive a safe empty state when the saved
  shown-to-players encounter cannot be restored safely.
- Recovery UI should stay quiet and GM-facing; no normal Player Display
  controls or diagnostics are added.

## Clarification Notes

- This feature follows F-007A and F-007B so it can harden the actual persisted
  storage shape and restore workflow.
- Planning pass on 2026-07-03 is documentation-only. It records recommended
  policy and test scope before implementation to keep the later coding session
  smaller.
- Recommended malformed-fog policy: do not fail the whole campaign if the
  invalid fog belongs to one encounter. Load the encounter without runtime fog,
  attach a GM-visible diagnostic to that encounter or campaign, and preserve the
  original `campaign.json` bytes until the GM makes an explicit successful
  mutation.
- Recommended missing-asset policy: keep the campaign and encounter listed,
  mark the affected map as unable to load, block or degrade `Show to Players`
  safely for that encounter, and avoid exposing broken image data to the Player
  Display.
- Recommended shown-to-players restore policy: if saved `activeMapId` points to
  an encounter with malformed fog, restore the encounter with no fog only if the
  map asset is valid and a GM diagnostic is available. If the saved
  shown-to-players encounter's map asset is missing or invalid, restore no
  Player Display encounter and report the problem to the GM.
- User decision on 2026-07-03: if the saved shown-to-players encounter has
  malformed fog but a valid map asset, restore it to the Player Display without
  that malformed fog and show the recovery diagnostic to the GM.
- User decision on 2026-07-03: missing map asset details should appear only in
  GM diagnostics, not directly on normal encounter cards.
- User decision on 2026-07-03: storage-format documentation should live in
  `docs/development.md`.
- Recovery is non-destructive by default. Reads, library listing, campaign open,
  and failed recovery must not rewrite `campaign.json`, delete assets, or strip
  unknown metadata.
- Keep F-007C inside the existing local JSON storage design. Do not introduce a
  database, migrations framework, cloud sync, map replacement flow, or
  persistence status UI.

## Planning Scope

Keep these slices inside F-007C unless implementation shows one slice is large

## Validation Plan

Use representative valid, malformed, mismatched, missing-file, and
partial-failure fixtures, then run automated and browser checks that cover
non-destructive recovery.

Planned fixture matrix:

| Case | Expected Recovery |
| --- | --- |
| Valid campaign with valid fog and assets | Opens normally, restores shown encounter and fog. |
| Campaign folder with malformed `campaign.json` | Valid campaigns remain listed; malformed campaign is skipped with existing non-destructive library diagnostic. |
| Encounter with non-array `fog` | Campaign opens where practical; affected encounter has no runtime fog and a GM-visible diagnostic; file is not rewritten. |
| Encounter with invalid fog operation type | Campaign opens where practical; affected encounter has no runtime fog and a GM-visible diagnostic; file is not rewritten. |
| Encounter with invalid fog rectangle bounds | Campaign opens where practical; affected encounter has no runtime fog and a GM-visible diagnostic; file is not rewritten. |
| One invalid fog encounter and one valid fog encounter | Valid encounter fog still hydrates; invalid encounter fog is isolated. |
| Saved shown encounter has invalid fog but valid asset | Player Display restores the shown encounter without malformed fog; GM diagnostics report the recovery. |
| Saved shown encounter has missing map asset | Player Display remains waiting or enters safe empty/error state; GM sees diagnostic; no asset path leaks to players. |
| Unshown encounter has missing map asset | Campaign remains usable; other encounters can still be shown; affected encounter reports map-load problem. |
| Unknown campaign, encounter, fog, or future metadata fields | Unknown metadata is preserved through later successful writes where current compatibility policy supports it. |
| Failed recovery followed by canceled or rejected mutation | `campaign.json`, map files, runtime state, and Player Display remain unchanged. |

Focused automated checks should include storage/schema tests for recovery
normalization, server route tests for campaign open and player projection
safety, and Chromium workflow tests for GM-visible diagnostics plus Player
Display waiting/error behavior. Manual device checks remain unnecessary unless
implementation changes local-network, HTTPS, or iPad-specific behavior.
