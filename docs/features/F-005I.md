## Preliminary Design-Language Audit

This clarification pass reviewed the current GM View and Player Display against
`docs/design-language.md`, `docs/ui-terminology.md`, implemented UI structure,
and existing Chromium workflow coverage.

### Principle Coverage

- The Map Is The Hero: encounter thumbnails are prominent, the Encounter
  Workspace map stage is larger than surrounding controls, and the Player
  Display gives the map the full remaining viewport.
- One Question Per Screen: the implemented navigation hierarchy maps cleanly to
  Campaign Library, Campaign, Encounter Workspace, and Player Display. Existing
  workflows assert one Back action on non-root GM screens and no Back action on
  the Campaign Library root. Campaign Library administration controls remain a
  review risk below.
- Calm Interface: encounter upload, rename, reorder, and delete controls are
  hidden in normal browsing and appear only after `Manage Encounters`. The
  visible Campaign Library create/edit/delete controls and workspace placeholder
  panel remain review risks below.
- Progressive Disclosure: existing Chromium coverage confirms Normal Mode hides
  encounter upload, rename, reorder, and delete controls while preserving
  browse, open, and `Show to Players` actions. Campaign Library administration
  is less progressively disclosed.
- Maps First: encounter cards use image thumbnails before administration, and
  workspace and player map surfaces are visually dominant.
- Never Surprise the GM: existing Chromium coverage confirms opening an
  encounter for prep does not change the Player Display or refetch the shown
  player asset. Editing campaign metadata, managing encounters, and deleting
  unshown content also preserve the Player Display in covered workflows.
- Editing Is Different From Running: Manage Mode separates encounter
  administration from normal browsing, while `Show to Players` remains
  available as the explicit running action. Campaign Library administration is
  not yet separated into its own mode.
- Game Concepts, Not Technical Concepts: normal UI copy uses Campaign,
  Encounter, Map, Player Display, `Show to Players`, and `Shown to Players`.
  Troubleshooting diagnostics are acceptable when data is invalid. Delete
  confirmation copy that says `map file` is a technical-language polish risk.
- Cards Over Tables: campaigns and encounters are card-based, with no table UI
  in the reviewed surfaces.
- Consistent Navigation: breadcrumbs reflect Campaign Library to Campaign to
  Encounter Workspace, and each non-root GM screen has one obvious Back action.
- Designed For A Standing GM: primary buttons have large targets, cards are
  browsable, and card/workspace show actions are reachable without precise
  table-style controls. Campaign Library card administration adds extra choices
  during normal browsing.
- Respect The Table: Player Display remains read-only, and player viewport
  controls stay local to that display. No tokens, dice, initiative, notes,
  dynamic lighting, or VTT-style controls appear.
- Prep Once, Run Smoothly: the current workflow supports Open Campaign, Open
  Encounter, and Show to Players with Player Display separation preserved.
- Encounters Are The Primary Unit: visible workflow centers Campaigns and
  Encounters while preserving maps as the visual artifact inside encounters.
- Rule of Three: the main GM hierarchy is three levels deep: Campaign Library,
  Campaign, and Encounter Workspace. Campaign cards currently expose more than
  three normal-mode actions when create/edit/delete are counted across the
  screen, which is captured as a Campaign Library administration risk.

### Review Risks And Recommendations

- Campaign Library administration controls are visible in normal browsing:
  `New campaign`, `Create`, campaign-card `Edit`, and campaign-card `Delete...`.
  This is broader than F-005M's empty-campaign delete affordance and conflicts
  with Progressive Disclosure more than the encounter gallery does.
  Recommendation: decide in a later reviewed feature whether Campaign Library
  needs a `Manage Campaigns` mode, or explicitly accept campaign creation and
  card editing/deletion as limited exceptions. Do not assign this to F-005J
  unless the change is purely visual and does not alter workflow.
- The Encounter Workspace shows a visible `Future Tools` reserved panel before
  those tools exist. F-005G validates that this panel is visually subordinate to
  the map and contains no fog controls, but it still competes with Calm
  Interface and One Question Per Screen more than an empty workspace needs to.
  Recommendation: feed this into F-005J if the change can remain visual polish,
  or defer it to F-006 when real fog tools replace the placeholder.
- `Shown to Players` is both a status phrase and, for the current shown
  encounter, an enabled clear action. F-005K records this as an intentional
  exception with explicit accessible names. Recommendation: keep this exception
  documented during F-005I and re-check that F-005J does not make the clear
  action look like passive status.
- The Player Display status bar exposes zoom and fit controls. This is allowed
  by the architecture as local display control, but it should remain visually
  subordinate to the map. Recommendation: include Player Display chrome density
  in F-005J's visual polish review without changing read-only behavior.
- Encounter delete confirmation copy says `map file`, which exposes an
  implementation-ish detail in an otherwise encounter-centered workflow.
  Recommendation: treat this as F-005J copy polish if changed, with no behavior
  or storage change.
