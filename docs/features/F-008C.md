# F-008C: Undo Fog Action

## Objective

Let the GM undo the most recent fog operation for the selected encounter.

## User Value

The GM can recover from an accidental hide, reveal, or clear during prep or play
without rebuilding fog manually or reloading campaign data.

## Non-Goals

- Do not add brush tools, circles, polygons, shape editing, or per-shape
  selection.
- Do not add redo.
- Do not add multi-user conflict handling.
- Do not add player controls.
- Do not change which encounter is `Shown to Players`.
- Do not add dynamic lighting, tokens, vision cones, notes, initiative, or other
  VTT features.
- Do not migrate storage/API terminology such as `maps`, `activeMapId`, or
  `fog`.

## Acceptance Notes

- Undo controls live in the Encounter Workspace Fog tools group.
- Undo is a compact text button labeled `Undo`.
- Undo affects only the selected encounter.
- Undo is disabled when the selected encounter has no undoable fog change.
- Undo can be clicked repeatedly; each click undoes the latest undoable fog
  change for the selected encounter.
- Undoing a Hide or Reveal operation removes the latest fog operation for the
  selected encounter.
- Undoing Clear Fog restores the selected encounter's prior ordered fog
  operations.
- Clear Fog remains confirmed before it clears fog.
- Undo respects the selected/editing versus `Shown to Players` distinction.
- If the selected encounter is currently `Shown to Players`, undo updates the
  Player Display live.
- If the selected encounter is not `Shown to Players`, undo does not change the
  Player Display.
- Undo autosaves the resulting fog state to local campaign storage.
- Undo history is runtime-only and does not survive campaign reloads or server
  restarts.
- Invalid or rejected undo requests leave runtime state, campaign storage, and
  Player Display unchanged.
- Failed undo shows quiet GM-local status text such as `Could not undo fog.
  Nothing changed.`
- Player Display remains read-only.

## Clarification Notes

- Scaffolded on 2026-07-04 as the next proposed F-008 quality-of-life slice.
- User confirmed on 2026-07-04 that undo should be single-step but repeatable:
  each click undoes the current latest undoable fog change.
- User confirmed on 2026-07-04 that Clear Fog should be undoable by restoring
  the selected encounter's prior ordered fog operation list.
- User confirmed on 2026-07-04 that undo history should not survive campaign
  reloads; only the resulting fog state persists.
- User confirmed on 2026-07-04 that Undo should be a compact text button in the
  Fog tools group.
- User confirmed on 2026-07-04 that undo failures should use quiet GM-local
  status text.
- Because fog now persists to `campaign.json`, undo must define how it interacts
  with autosave before implementation begins. The clarified behavior is that
  each successful undo autosaves the resulting fog state, while the undo history
  itself remains runtime-only.

## Harness Compliance

Harness Version: 1

## SDLC Tier

Tier: 3.

Rationale: F-008C is expected to touch fog persistence, filesystem writes,
server state, Socket.IO synchronization, Player Display projection, player
read-only guarantees, and map/fog rendering semantics. Those areas are Tier 3
under `docs/decisions/decision-004-mandatory-subagent-sdlc.md`.

## Affected Systems

- GM view: adds an Undo fog control in the Encounter Workspace.
- Player view: receives live fog updates only when the selected encounter is
  also `Shown to Players`.
- Local server: likely adds a GM-only undo mutation for selected encounter fog
  state.
- Map/fog rendering: re-renders the ordered fog operation list after undo.
- Storage: autosaves the resulting fog state in local campaign storage.
- Documentation: acceptance tests, README current status, roadmap, feature
  tracker, and possibly development storage notes after implementation.

## Validation Plan

Define automated coverage after clarification. Expected coverage should include
undoing Hide, undoing Reveal, selected-unshown isolation, shown-player live
updates, autosave persistence, rejected undo atomicity, Player Display
read-only behavior, repeated undo clicks, runtime-only undo history after
reload, quiet failure messaging, and Clear Fog undo behavior.

Run focused Node and Chromium tests for the changed workflow, then run
`npm run harness:check`, `npm run test:browser`, and `npm run quality` before
marking the feature complete.
