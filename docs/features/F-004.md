# F-004: Campaign and Map Library

## Objective

Allow the GM to create or open a campaign shell and manage multiple maps inside it.

## User Value

The GM can prepare the 1 to 3 encounters they expect for a session, come back days later, add more maps, rename maps, reorder maps, and keep maps grouped by campaign.

## Non-Goals

- Do not add tokens.
- Do not add fog controls.
- Do not add initiative tracking.
- Do not add notes.
- Do not add map replacement.
- Do not add multiple fog states per map.
- Do not add cloud storage or user accounts.

## Acceptance Notes

- GM can create a campaign.
- GM can open an existing campaign.
- Campaigns are stored locally as folders.
- GM can upload or add maps into a campaign.
- The original file name is preserved as the default map name.
- GM can rename maps later.
- GM can reorder maps manually.
- GM can select an active map.
- Player view displays the active map.
- Player view remains read-only.
- No fog controls are exposed yet.

## Suggested Storage Direction

Campaigns should use local, inspectable files. A migration-friendly first shape is:

```text
tabletopfog-data/
  The Long Walk/
    campaign.json
    maps/
      forest-ambush.png
      goblin-cave.png
```

Example `campaign.json`:

```json
{
  "name": "The Long Walk",
  "activeMapId": "forest-ambush",
  "maps": [
    {
      "id": "forest-ambush",
      "name": "Forest Ambush",
      "file": "maps/forest-ambush.png",
      "order": 1,
      "fog": []
    }
  ]
}
```

## Affected Systems

- GM view: campaign open/create, map add, rename, reorder, and active-map selection controls.
- Player view: read-only display of the active map.
- Local server: serves campaign metadata and selected map assets from local folders.
- Map/fog rendering: static active-map rendering only.
- Storage: local campaign folder, map files, and `campaign.json`.
- Documentation: update acceptance tests and storage-format notes.

## Validation Plan

Create a campaign, add two representative map images, rename and reorder them, select each as active, and verify the player view displays the selected active map without exposing mutation controls.

## Open Questions

- Where should `tabletopfog-data/` live by default on Chromebook and MacBook?
- Should opening a campaign use a path typed into the app, a configured data root, or a browser file-picker flow?
- How should filename collisions be handled when adding maps with the same original file name?
- What characters should be allowed or normalized in campaign folder names and map file names?
- Should manual map ordering be stored only as explicit `order` numbers or as array order plus stable IDs?
