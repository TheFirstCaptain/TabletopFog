# F-004: Campaign and Map Library

## Objective

Allow the GM to create or open a campaign shell and manage multiple maps inside it.

## User Value

The GM can prepare the 1 to 3 encounters they expect for a session, come back days later, add more maps, rename maps, reorder maps, and keep maps grouped by campaign.

## Non-Goals

- Do not add tokens.
- Do not add fog controls.
- Do not add initiative tracking.
- Do not add notes.
- Do not add map replacement.
- Do not add multiple fog states per map.
- Do not add cloud storage or user accounts.

## Acceptance Notes

- GM view starts at a campaign library page.
- The campaign library lists campaigns discovered from the default data root.
- GM can create a campaign.
- GM can open an existing campaign.
- Campaigns are stored locally as folders.
- GM can upload or add maps into a campaign.
- Map upload accepts validated PNG, JPEG, GIF, and WebP files up to 100 MB.
- Rejected uploads do not add a map file or campaign metadata.
- The original file name is preserved as the default map name.
- GM can rename maps later.
- GM can reorder maps manually.
- GM can select an active map.
- Player view displays the active map.
- Player view remains read-only.
- No fog controls are exposed yet.

## Suggested Storage Direction

Campaigns should use local, inspectable files. The default data root should be:

```text
~/TabletopFog/tabletopfog-data/
```

This keeps campaign data outside the repository while staying easy to find, inspect, back up, and copy on both MacBook and Chromebook Linux.

Implemented override:

```sh
TABLETOPFOG_DATA_DIR=/path/to/tabletopfog-data npm run local
```

A migration-friendly first shape is:

```text
~/TabletopFog/tabletopfog-data/
  The Long Walk/
    campaign.json
    maps/
      forest-ambush.png
      goblin-cave.png
```

Example `campaign.json`:

```json
{
  "name": "The Long Walk",
  "activeMapId": "forest-ambush",
  "maps": [
    {
      "id": "forest-ambush",
      "name": "Forest Ambush",
      "file": "maps/forest-ambush.png",
      "order": 1,
      "fog": []
    }
  ]
}
```

## Campaign Opening Model

The GM view should start at a campaign library page. The local Node server should scan the default data root and list each subfolder that contains a valid `campaign.json`.

For V1:

- GM opens `/gm` and sees the campaign library.
- GM can create a new campaign from the library.
- GM can open one existing campaign at a time.
- Do not require the GM to browse arbitrary folders every time.
- Do not add a general folder picker in V1.
- The data root can be overridden with `TABLETOPFOG_DATA_DIR`.

## Map File Collision Policy

When adding maps, preserve the original uploaded file name in metadata and derive the default display name from it. Stored map asset paths must still be unique within the campaign `maps/` folder.

For V1:

- If no collision exists, keep the original file name for the stored map asset.
- If a collision exists, append `-2`, `-3`, and so on before the extension.
- Preserve the original uploaded filename as `originalFileName`.
- Default the map display name to the original filename without its extension.
- Do not block a map upload only because another map file has the same original name.

Example:

```json
{
  "id": "forest-ambush-2",
  "name": "forest-ambush",
  "originalFileName": "forest-ambush.png",
  "file": "maps/forest-ambush-2.png",
  "order": 2,
  "fog": []
}
```

## Filesystem Name Policy

Campaign folder names and stored map asset names should use filesystem-safe path segments. User-facing display names can remain more flexible because they are stored in `campaign.json` and are not used as paths.

For V1 filesystem names:

- Allow letters, numbers, spaces, hyphens, and underscores.
- Replace other characters with `-`.
- Collapse repeated `-` characters.
- Trim leading and trailing spaces, dots, hyphens, and underscores.
- Reject or require a different name if normalization produces an empty name.
- Treat case-only duplicates as collisions because filesystem case sensitivity differs by host.

Examples:

```text
The Long Walk      -> The Long Walk
Goblin Cave #2.png -> Goblin Cave-2.png
boss/finale.png    -> boss-finale.png
..secret..png      -> secret.png
```

## Manual Ordering Policy

Map ordering should use both array order and an explicit integer `order` field, with `order` treated as authoritative.

For V1:

- Store maps in sorted order for human readability.
- Include an explicit integer `order` on each map.
- On load, sort by `order`, then by array position as a fallback.
- On manual reorder, rewrite clean sequential order values: `1`, `2`, `3`, and so on.
- If duplicate or missing order values are found, repair them on the next save.
- Up/down controls and future drag/drop controls should update the same order representation.

## Affected Systems

- GM view: campaign open/create, map add, rename, reorder, and active-map selection controls.
- Player view: read-only display of the active map.
- Local server: serves campaign metadata and selected map assets from local folders.
- Map/fog rendering: static active-map rendering only.
- Storage: local campaign folder, map files, and `campaign.json`.
- Documentation: update acceptance tests and storage-format notes.

Create a campaign, add two representative map images, rename and reorder them, select each as active, and verify the player view displays the selected active map without exposing mutation controls.

## SDLC Stage Outputs

### Clarification

- Default data root is `~/TabletopFog/tabletopfog-data/`.
- `/gm` starts at a campaign library page.
- The server scans the default data root and lists folders with valid `campaign.json`.
- GM can create and open one campaign at a time.
- No general folder picker is included in V1.
- Stored map file names are normalized and collision-safe.
- Original uploaded file names are preserved in metadata.
- Display names are separate from filesystem paths and can be renamed later.
- Manual ordering uses sorted array order plus authoritative integer `order` values.

### Architecture Review

- Filesystem access stays server-side; browser code uses local API routes.
- Route-based roles remain appropriate: `/gm` mutates state and `/player` renders read-only state.
- Campaign folders with `campaign.json` and `maps/` are accepted as local, inspectable, migration-friendly storage.
- F-004 owns minimal active-map display so the player can see the selected campaign map. Richer viewport polish remains available for F-005.
- Scope remains limited to local campaign/map prep buckets. No fog controls, tokens, notes, initiative, accounts, cloud sync, or arbitrary folder picker.

### Implementation Design

- Use an injectable campaign storage module so automated tests can use temporary data roots.
- Use API endpoints for campaign list/create/open, map add/rename/reorder, active-map selection, and map asset serving.
- Use Socket.IO `state:sync` for active campaign/map updates to connected GM and player views.
- Use raw binary map upload from the browser instead of adding a multipart dependency.
- Validate image signatures, filename extensions, content types, and the 100 MB
  limit at the server storage boundary before writing files.
- Write `campaign.json` atomically through a temporary file and rename.
- Keep campaign list and open operations non-mutating; preserve unrecognized
  campaign and map fields during later explicit writes.
- Do not delete existing user campaign data during failed normalization, load, or save operations.

### Test and Validation Design

- Add storage tests for safe names, case-insensitive collisions, campaign discovery, campaign creation, map filename collisions, order repair, rename, active map persistence, and reload behavior.
- Add server tests for API lifecycle, active-map Socket.IO sync, map asset serving, and player read-only controls.
- Keep tests on Node's built-in test runner and temporary filesystem roots.
- Manual validation should create `The Long Walk`, add duplicate-named map files, rename and reorder maps, select each active map, and confirm `/player` displays the selected active map without mutation controls.

### UX and Workflow Review

- GM workflow starts with a campaign library, then an open-campaign map manager.
- Map management stays list-first: add map, rename, move up/down, and show to players.
- Use display-oriented labels such as `Show to players`; avoid VTT-like labels such as scene, encounter, deploy, or publish.
- Player view renders only the active map plus minimal loading/error state.
- Empty campaign, no active map, failed image load, and disconnected socket states should be visible without adding GM-only controls to the player view.

  validation.

### Accepted MVP Security and Restore Boundaries

- GM-only API writes use the existing trusted-LAN route/referer model from the MVP role decision. This prevents normal `/player` page interactions from mutating state, but it is not authentication and does not protect against a deliberate same-LAN client spoofing GM requests.
- The player receives only active-map display state and can fetch only the current active map through `/api/player/active-map/asset`.
- After a server restart, the player view waits until the GM opens a campaign. F-004 persists campaign metadata and active map in `campaign.json`, but it does not auto-open the last campaign on server startup.
