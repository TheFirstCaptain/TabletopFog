# F-005A: Campaign Landing Page Polish

## Objective

Replace the utilitarian campaign list with a more polished campaign landing page using cards.

## User Value

The GM starts from a lightweight fantasy campaign shelf that is easier to scan before a session while keeping the app focused on local prep and table display.

## Non-Goals

- Do not add player or member management.
- Do not add campaign notes.
- Do not add character sheets.
- Do not add cloud sync.
- Do not add decorative image assets unless explicitly approved.
- Do not redesign storage beyond metadata needed for campaign description or icon.
- Do not add fog controls.
- Do not add player tokens.
- Do not add VTT-style navigation.

## Acceptance Notes

- GM `/gm` starts on a campaign landing page.
- Campaigns display as cards, not a plain list.
- Each campaign card shows the campaign name.
- Each campaign card shows a short description when available.
- Each campaign card shows a small campaign icon when available.
- Campaign cards may show basic metadata such as map or encounter count, or last modified, only when available without heavy new work.
- Clicking a campaign card opens that campaign.
- New campaign creation remains available.
- Campaign search or filtering may be included only if lightweight; otherwise document it as future polish.
- The page keeps the F-004A fantasy/parchment visual theme.
- The layout works on Chromebook-sized screens.
- Invalid campaign diagnostics remain visible and usable.

## Terminology Notes

- Use "campaign" for the top-level local folder and prep bucket.
- Prefer lightweight fantasy-tabletop language over full-VTT concepts such as parties, members, characters, or campaign dashboards.
- Any new campaign description or icon metadata must preserve compatibility with existing campaign folders.

## Affected Systems

- GM view: campaign landing page, campaign cards, create-campaign entry point, invalid campaign diagnostics.
- Player view: not affected.
- Local server: may expose existing or lightweight campaign card metadata.
- Map/fog rendering: not affected.
- Storage: only optional campaign description or icon metadata if approved during planning.
- Documentation: feature tracker, roadmap, acceptance tests, README if current status or workflow changes.

## Validation Plan

Open `/gm` with valid campaigns, no campaigns, and at least one invalid campaign folder. Confirm campaign cards are readable, open the intended campaign, preserve create-campaign access, retain invalid-folder diagnostics, and fit on Chromebook-sized viewports without adding out-of-scope VTT controls.

## Open Questions

- Should campaign icons be uploaded images, selected built-in symbols, emoji, or deferred?
- Should campaign descriptions be editable in F-005A or planned as a separate follow-up?
- Should search be included now or deferred?

## Status

Proposed.
