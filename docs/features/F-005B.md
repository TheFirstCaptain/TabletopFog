# F-005B: Encounter Card Gallery and Workspace Entry

## Objective

Replace the open-campaign map manager list with a visual encounter gallery and introduce the concept of opening an encounter workspace.

## User Value

The GM can browse prepared encounter maps visually, open one for prep, and deliberately decide when a map should appear on the player display.

## Non-Goals

- Do not add initiative tracking.
- Do not add notes.
- Do not add tokens.
- Do not add dynamic lighting.
- Do not add VTT-style left navigation.
- Do not add campaign members or characters.
- Do not add fog controls except for placeholders needed to prepare F-006.
- Do not rename storage or APIs without a reviewed migration plan.

## Acceptance Notes

- Open campaign page shows encounter/map cards with map thumbnails.
- Cards display the encounter/map name.
- Cards may display simple status such as `Shown to Players` if that map is currently on the player display.
- The map thumbnail is visually prominent because the map is the primary artifact.
- GM can still add or upload maps.
- GM can still rename maps.
- GM can still reorder maps.
- Clicking an encounter/map card opens an encounter workspace or workspace placeholder.
- Clicking an encounter/map card does not automatically change the player display.
- Showing a map to players is an explicit action such as `Show to Players`.
- The player display changes only after the explicit show action.
- The current player-shown encounter is visually distinguished.
- The layout preserves the fantasy/parchment theme and remains usable on Chromebook.

## Terminology Notes

- Prefer "encounter" in user-facing UI for prepared map cards.
- Preserve existing storage compatibility where needed; current storage may still use `maps`.
- Distinguish the selected/editing encounter from the shown-to-players encounter.
- Opening an encounter for editing is not the same as showing it to players.
- Existing `activeMapId` storage/API terminology currently represents the player-shown map, not necessarily the GM-selected editing encounter.
- A future migration to clearer naming such as `shownMapId` may be considered, but this feature should not perform a risky storage migration without explicit scope.

## Affected Systems

- GM view: open-campaign encounter gallery, thumbnail cards, workspace entry, explicit show action.
- Player view: updates only when the GM explicitly shows an encounter to players.
- Local server: may continue using existing active-map endpoints until a reviewed rename or migration is planned.
- Map/fog rendering: thumbnail display and existing canvas display paths.
- Storage: existing map metadata remains compatible.
- Documentation: update acceptance tests and terminology notes around selected/editing versus shown-to-players encounters.

## Validation Plan

Open a campaign with multiple maps, verify encounter cards with thumbnails, rename and reorder entries, open an encounter workspace without changing the player display, then use `Show to Players` and confirm only that explicit action updates the player view.

## Open Questions

- Should `Show to Players` live on each encounter card, inside the workspace, or both?
- Should opening the workspace be an in-page view, a route, or a modal?
- What should the player display show when no encounter has been explicitly shown?
