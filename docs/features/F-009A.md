# F-009A: Local Quality Command and Baseline Enforcement

## Objective

Provide one documented local command that runs the repository's required
automated quality checks with deterministic pass/fail behavior.

## User Value

Maintainers and agents can verify a change using one repeatable command instead
of relying on memory or prose-only completion claims.

## Non-Goals

- Do not add continuous integration; F-009C owns remote enforcement.
- Do not refactor application modules to satisfy new checks.
- Do not introduce arbitrary coverage or file-size targets without baselines.
- Do not add a frontend application build system.

## Acceptance Notes

- `npm run quality` or an equivalently clear command is the authoritative local
  validation entry point.
- The command runs automated tests, linting, formatting checks, dependency
  auditing, coverage measurement/regression policy, and module-growth checks.
- Failures identify the failed check and return a non-zero status.
- Dependency auditing uses a documented blocking severity and exception policy.
- Coverage and module checks use recorded baselines and ratchet against
  regression.
- Existing documented development commands remain available.

## Clarification Notes

- E-005 must record the module baseline before file-growth enforcement is set.
- Use ESLint flat configuration with environments for CommonJS Node code and
  plain browser scripts.
- Use Prettier for JS, JSON, CSS, HTML, and YAML. Exclude Markdown initially.
- Use Node's built-in test coverage and set initial thresholds from the measured
  baseline, with no-regression ratcheting rather than aspirational targets.
- High and critical `npm audit` findings block quality. Moderate findings remain
  visible but initially non-blocking.
- Treat registry or network failure during dependency audit as a quality failure.
- Every F-009A check blocks `npm run quality`; file-scoped optimization is
  deferred until a demonstrated performance need exists.
- Preserve existing `npm test`, startup, and certificate commands.

## SDLC Agent Outputs

### Architecture Review

- Use a versioned machine-readable module baseline covering every JavaScript file
  under `server/`, `public/`, and `scripts/`.
- Use separate ESLint flat-config entries for CommonJS Node code and classic
  browser scripts with the read-only Socket.IO `io` global.
- Keep F-009A behavior-preserving and preserve existing startup, certificate,
  and test commands.
- Scope Node coverage to `server/**/*.js` and `scripts/**/*.js`; F-009D owns
  browser execution and coverage.
- Use whole-number floors from a successful measured baseline to reduce
  Node/V8 patch-level noise.

### Implementation Design

- Add ESLint, `@eslint/js`, `globals`, and Prettier as development dependencies.
- Add explicit `lint`, `format`, `format:check`, `modules:check`,
  `test:coverage`, `audit:high`, and `quality` scripts.
- Implement a small testable quality runner that executes read-only stages in
  this order: lint, format check, module check, coverage/tests, dependency audit.
- Implement a testable module-baseline checker using repository-relative POSIX
  paths and physical line counts.
- Fail on both growth and shrinkage. Growth requires architecture disposition;
  shrinkage requires lowering the recorded baseline so the ratchet remains
  durable.
- Keep audit last because it is network-dependent. Do not add automatic audit
  fixes or silent exceptions.

### Test and Validation Design

- Add red/green unit tests for exact baseline success, growth, shrinkage,
  missing/stale entries, unrecorded modules, and empty responsibilities.
- Add red/green unit tests proving the quality runner executes stages in order,
  stops on failure, preserves the exit status, and identifies the targeted
  rerun command.
- Validate each quality subcommand independently and through the aggregate
  command, including known failing cases where practical.
- Validate `npm ci` followed by `npm run quality` from a clean dependency state.
- Do not retain an intentionally vulnerable dependency for audit testing; test
  orchestration failure with injection and validate the real current audit.
- A subagent coverage attempt was invalid because its restricted environment
  denied localhost binding. The main-agent run with localhost permission passed
  33/33 tests and measured 76.68% lines, 69.62% branches, and 81.58% functions
  for Node production code. Initial enforced floors are 76%, 69%, and 81%.

### UX and Workflow Review

- Keep `npm run quality` authoritative and read-only; retain granular commands
  for targeted iteration and a separate explicit `npm run format` writer.
- Print a concise stage label and exact rerun command on failure.
- Fail fast and place network-dependent audit last.
- Document that audit network failure blocks full quality; do not provide an
  offline bypass that can be mistaken for completion.
- Treat module-ratchet failure as a review trigger, not proof that a larger file
  is inherently defective.

### Code Review

- Blocking review finding: ESLint 10 did not support the full documented Node 22
  line. Resolved by using ESLint and `@eslint/js` 9.39.4.
- Blocking review finding: built-in coverage threshold flags require Node 22.8.0
  or newer. Resolved by making the package and documentation minimum precise.
- Medium review finding: editable baseline roots could bypass `public/` or
  `scripts/` discovery. Resolved by enforcing the exact required roots in code
  and adding a bypass regression test.
- Medium review finding: required quality-stage tests derived expectations from
  the implementation. Resolved by pinning the literal five-stage list and
  testing failure at every stage.
- Medium review finding: temporary exception metadata was not validated.
  Resolved by requiring an `E-NNN` identifier and non-empty reason with focused
  coverage.
- Independent architecture and test re-reviews found no remaining blockers.
- Accepted low residual: real negative lint, format, coverage, and registry
  failures are validated through tool behavior and injected orchestration rather
  than retained destructive or network-dependent fixtures.
