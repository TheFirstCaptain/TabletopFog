# F-006: Manual Fog of War

## Objective

Add manual hide/reveal fog controls to the encounter workspace.

## User Value

The GM can prep fog on a selected encounter map, explicitly show that encounter to players, and then reveal areas during play without turning TabletopFog into a full VTT.

## Non-Goals

- Do not start maps under a full black fog layer.
- Do not add dynamic lighting.
- Do not add tokens or vision cones.
- Do not add automation based on rules, NPCs, or character stats.
- Do not add secret room layers.
- Do not add multiple fog states per encounter.
- Do not add initiative tracking.
- Do not add notes.
- Do not migrate storage or rename existing APIs unless explicitly approved in this feature's reviewed scope.

## Acceptance Notes

- [F-006A](./F-006A.md) tracks the first fog-adjacent workspace tools: GM-local
  zoom, a 5 ft grid overlay, and grid lock/unlock before manual fog drawing.
- [F-006B](./F-006B.md) establishes in-memory fog state, V1 operation-list
  semantics, and shared fog rendering without GM drawing tools.
- [F-006C](./F-006C.md) adds the rectangle Hide tool.
- [F-006D](./F-006D.md) adds the rectangle Reveal tool.
- [F-006E](./F-006E.md) adds an explicit Clear Fog control.
- [F-006F](./F-006F.md) finishes fog tool polish, edge-case validation, and
  umbrella completion review.
- Fog tools live in the encounter workspace.
- The GM edits fog on the selected/editing encounter.
- Fog state belongs to that encounter/map.
- Maps start visible.
- GM can add fog or hide areas using simple shapes such as rectangles or circles.
- GM can remove or reveal fog from hidden areas during play.
- Player display receives fog updates only for the encounter currently shown to players.
- If the GM edits fog on an encounter that is not currently shown to players, the player display does not change.
- If the GM edits fog on the encounter currently shown to players, the player display updates live.
- GM view shows fog semi-transparently unless implementation review chooses a better accessible treatment.
- Player view shows fog opaquely or near-opaquely.
- Hidden areas remain hidden on the player display.
- Player view remains read-only.
- No multiple fog states per encounter.
- No dynamic lighting.
- No tokens or vision cones.

## Child Feature Sequence

F-006 is intentionally split into small child features so manual fog can become
usable incrementally while keeping review scope narrow.

| ID | Title | Purpose |
| --- | --- | --- |
| F-006A | GM Map Zoom and Grid Calibration | Completed setup work for map alignment before drawing fog. |
| F-006B | Fog State and Rendering Foundation | Establish encounter-scoped in-memory fog operations and render fog correctly in GM View and Player Display. |
| F-006C | Rectangle Hide Tool | Let the GM draw rectangular hidden areas on the selected encounter. |
| F-006D | Rectangle Reveal Tool | Let the GM draw rectangular reveal operations that cut visibility back through existing fog. |
| F-006E | Clear Fog | Let the GM explicitly clear all fog for the selected encounter. |
| F-006F | Fog Tool Polish and Validation | Tighten workflow, edge cases, responsive behavior, and umbrella validation before marking F-006 done. |

The V1 fog model is an ordered operation list per encounter. Hide and reveal
operations are replayed in order during rendering. A reveal operation visually
removes fog from its rectangle, even when it cuts through an earlier hide
operation. A later hide operation can cover that area again. This avoids early
shape-normalization complexity while preserving intuitive table behavior.

## Terminology Notes

- The selected/editing encounter is the encounter open in the GM workspace.
- The shown-to-players encounter is the encounter currently displayed on `/player`.
- Opening an encounter workspace is not the same as showing it to players.
- `Show to Players` remains an explicit GM action.
- Existing storage/API names such as `maps` and `activeMapId` may remain during implementation for compatibility.
- Where existing `activeMapId` appears, document and treat it as the player-shown map, not necessarily the GM-selected editing encounter.
- Consider a future reviewed migration to clearer terminology such as `shownMapId`, but do not bundle that migration into fog work without explicit approval.

## Affected Systems

- GM view: hide and reveal tools inside the encounter workspace for the selected/editing encounter.
- Player view: read-only rendering with hidden areas obscured only for the shown-to-players encounter.
- Local server: syncs fog state for the shown-to-players encounter when relevant and preserves separation from GM workspace selection.
- Map/fog rendering: visible map plus fog shapes or mask, with GM/player opacity differences.
- Storage: in-memory fog state until F-007 persistence unless F-006 explicitly expands storage scope.
- Documentation: update fog correctness tests, terminology notes, and storage-shape notes.

## Validation Plan

Start from a visible encounter map in the GM workspace. Add simple fog shapes to an encounter not shown to players and confirm the player display does not change. Show that encounter to players, add and reveal fog, and confirm the player display updates live with hidden areas obscured. Repeat with another selected/editing encounter while the first remains shown to players.

## Open Questions

- Resolved for V1: use an ordered in-memory operation list rather than
  normalized shape state or a rasterized mask.
- Resolved for V1: GM view shows fog semi-transparently while player view shows
  it opaque or near-opaque.
- Resolved for V1: reveal operations replay after hide operations and cut
  through visible fog in their rectangle; later hide operations may cover the
  same area again.
- Resolved for V1: fog remains in-memory until F-007 persistence work.
- Should any storage/API terminology cleanup be split into a separate feature before persistence?
