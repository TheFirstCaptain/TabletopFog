# F-007A: Persist Per-Encounter Fog State

## Objective

Persist each encounter's ordered fog operations in the local campaign file so GM
prep survives campaign reloads without adding new UI.

## User Value

The GM can prepare or adjust fog, close or reopen a campaign, and keep that fog
work without relying on memory-only session state.

## Non-Goals

- Do not add cloud storage.
- Do not add user accounts.
- Do not add explicit save buttons or persistence status UI.
- Do not add campaign locking.
- Do not change shown-to-players restore behavior beyond preserving existing
  `activeMapId` metadata.
- Do not add missing-asset recovery UI.
- Do not add undo, brush tools, or multiple fog states.
- Do not migrate `activeMapId`, `maps`, or storage/API terminology.

## Acceptance Notes

- Fog append autosaves the selected encounter's ordered fog operations to
  `campaign.json`.
- Clear Fog autosaves an empty fog operation list for the selected encounter.
- Reopening a campaign hydrates each encounter's persisted fog into the existing
  runtime `fogOperations` state.
- Persisted fog remains scoped per campaign and encounter/map.
- Invalid fog append or clear requests remain non-mutating in memory and on
  disk.
- Existing campaign metadata, map order, display names, asset paths,
  `activeMapId`, and unknown forward-compatible fields are preserved.
- No new UI, route path, Socket.IO event name, storage terminology migration, or
  cloud/account behavior is introduced.

## Clarification Notes

- Autosave on each successful fog append or clear is the default F-007A
  behavior; there is no explicit save button in this slice.
- The existing map-level `fog` field in `campaign.json` remains the persisted
  storage field for ordered rectangle operations.
- Runtime APIs continue using `fogOperations`; storage compatibility names are
  not migrated in this feature.
- F-007A restores persisted fog when the GM opens a campaign. F-007B owns
  broader shown-to-players restore workflow after process restart.
- F-007C owns malformed persisted fog and missing-asset recovery details.

## Affected Systems

- GM view: existing fog actions become durable after campaign reopen.
- Player view: existing projected fog may render hydrated operations after the
  GM opens a campaign.
- Local server: fog append and clear routes coordinate with persistent campaign
  storage.
- Map/fog rendering: no rendering semantics change; hydrated operations use the
  existing renderer.
- Storage: map-level `fog` arrays in `campaign.json` persist ordered operations.
- Documentation: feature tracker, F-007 split, storage-format notes, and
  acceptance tests may need updates.
