# F-007B: Restore Shown-to-Players State

## Objective

Restore the saved shown-to-players encounter and its persisted fog state after a
campaign is reopened by the GM.

## User Value

The GM can resume a table session without manually re-showing the encounter or
reconstructing what the Player Display should see.

## Non-Goals

- Do not add cloud storage or accounts.
- Do not add multiple campaigns open at once.
- Do not add persistence status UI.
- Do not add missing-asset recovery UI.
- Do not auto-open the last campaign on server startup.
- Do not migrate `activeMapId` or storage/API terminology.

## Acceptance Notes

- Reopening a campaign with a saved `activeMapId` restores that encounter as
  shown to players.
- Restore is triggered by the GM opening a campaign through the existing
  campaign-open workflow.
- Server startup alone does not reopen a previous campaign or change Player
  Display state.
- Player Display receives only the restored shown-to-players encounter and its
  fog operations.
- GM selected/editing encounter state remains distinct from shown-to-players
  state.
- Opening an encounter still does not change the Player Display.
- No new cloud, account, token, notes, initiative, or VTT behavior is added.

## Clarification Notes

- This feature depends on F-007A persisted per-encounter fog.
- `activeMapId` remains the compatibility field for the encounter currently
  shown to players.
- User decision on 2026-07-03: F-007B should restore shown-to-players state
  only after the GM opens a campaign. It should not add automatic last-campaign
  startup restore.
- Rationale: GM-initiated campaign open avoids new last-open-campaign storage,
  avoids surprising the GM, and preserves the existing explicit local workflow.
