# F-007: Save and Load Campaign State

## Objective

Persist campaigns, maps, ordering, active map, and fog state locally.

## User Value

The GM can prepare campaigns over multiple days, resume existing campaign work, and keep fog prep for each map without relying on cloud storage.

## Non-Goals

- Do not add cloud storage.
- Do not add user accounts.
- Do not add multiple campaigns open at once.
- Do not add notes.
- Do not add map replacement.
- Do not add multiple fog states per map.

## Acceptance Notes

- Save campaign metadata.
- Save map list and ordering.
- Save map display names.
- Save active map.
- Save fog state per map.
- Reload a campaign later.
- Restored player view matches saved active map and fog state.
- Storage format is documented well enough to migrate later.
- No cloud storage.
- No user accounts.

## Affected Systems

- GM view: save/load campaign workflow and persistence status.
- Player view: displays restored active map and fog state.
- Local server: reads and writes campaign metadata and serves local map assets.
- Map/fog rendering: restores active map and per-map fog state.
- Storage: local campaign folders with inspectable files.
- Documentation: document storage format and migration expectations.

## Validation Plan

Create a campaign with multiple maps, rename/reorder maps, select an active map, add fog to at least one map, save, restart or reload the app, reopen the campaign, and verify GM/player views match saved active map and fog state.

## Open Questions

- Should persistence be explicit save, autosave, or both?
- Should the campaign folder be locked while the server is running?
- What version field should be included in `campaign.json` before first implementation?
- How should missing map files be reported when reopening a campaign?

## Status
