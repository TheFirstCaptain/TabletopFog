# F-007: Save and Load Campaign State

## Objective

Complete campaign persistence for fog state and later restore workflows after F-004's basic campaign/map metadata persistence exists.

## User Value

The GM can prepare fog over multiple days, resume existing campaign work, and restore player-visible state without relying on cloud storage.

## Non-Goals

- Do not add cloud storage.
- Do not add user accounts.
- Do not add multiple campaigns open at once.
- Do not add notes.
- Do not add map replacement.
- Do not add multiple fog states per encounter.
- Do not migrate `activeMapId` or other storage/API terminology without a reviewed migration plan.

## Acceptance Notes

- Preserve F-004 campaign metadata, map list, ordering, map display names, and player-shown map behavior.
- Save fog state per encounter/map.
- Reload a campaign later.
- Restored player view matches the saved shown-to-players encounter and fog state.
- Storage format is documented well enough to migrate later.
- No cloud storage.
- No user accounts.

## Terminology Notes

- F-005B and later distinguish the selected/editing encounter from the shown-to-players encounter.
- Existing `activeMapId` storage/API terminology may still represent the shown-to-players map until a reviewed migration chooses clearer naming such as `shownMapId`.
- Persistence must not collapse GM workspace selection and player-shown state into one ambiguous field.

## Affected Systems

- GM view: campaign reload workflow, fog persistence, and persistence status.
- Player view: displays restored shown-to-players encounter and fog state.
- Local server: reads and writes campaign metadata and serves local map assets.
- Map/fog rendering: restores shown-to-players encounter and per-encounter fog state.
- Storage: extends F-004 local campaign folders with persisted fog state.
- Documentation: document storage format and migration expectations.

## Validation Plan

Create a campaign with multiple encounters/maps, show one encounter to players, edit fog on at least one encounter, save or autosave according to the chosen workflow, restart or reload the app, reopen the campaign, and verify GM/player views restore the shown-to-players encounter and its fog state without confusing it with the GM selected/editing encounter.

## Open Questions

- Should persistence be explicit save, autosave, or both?
- Should the campaign folder be locked while the server is running?
- How should missing map files be reported when reopening a campaign?
