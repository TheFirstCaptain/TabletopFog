# F-007: Save and Load Campaign State

## Objective

Complete campaign persistence for fog state and later restore workflows after F-004's basic campaign/map metadata persistence exists.

## User Value

The GM can prepare fog over multiple days, resume existing campaign work, and restore player-visible state without relying on cloud storage.

## Non-Goals

- Do not add cloud storage.
- Do not add user accounts.
- Do not add multiple campaigns open at once.
- Do not add notes.
- Do not add map replacement.
- Do not add multiple fog states per map.

## Acceptance Notes

- Preserve F-004 campaign metadata, map list, ordering, map display names, and active map behavior.
- Save fog state per map.
- Reload a campaign later.
- Restored player view matches saved active map and fog state.
- Storage format is documented well enough to migrate later.
- No cloud storage.
- No user accounts.

## Affected Systems

- GM view: campaign reload workflow, fog persistence, and persistence status.
- Player view: displays restored active map and fog state.
- Local server: reads and writes campaign metadata and serves local map assets.
- Map/fog rendering: restores active map and per-map fog state.
- Storage: extends F-004 local campaign folders with persisted fog state.
- Documentation: document storage format and migration expectations.

## Validation Plan

Create a campaign with multiple maps, add fog to at least one map, save or autosave according to the chosen workflow, restart or reload the app, reopen the campaign, and verify GM/player views match saved active map and fog state.

## Open Questions

- Should persistence be explicit save, autosave, or both?
- Should the campaign folder be locked while the server is running?
- How should missing map files be reported when reopening a campaign?
