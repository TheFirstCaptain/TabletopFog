# F-009: Engineering Quality Gates

## Objective

Add automated repository checks that consistently enforce the existing test,
style, dependency-security, and maintainability expectations.

## User Value

Maintainers receive fast, repeatable feedback before regressions or unsafe
dependencies reach table use. Agents can verify repository rules instead of
relying only on handoff claims.

## Non-Goals

- Do not change product behavior.
- Do not add a frontend framework or application build system.
- Do not require cloud-hosted runtime services.
- Do not set arbitrary quality thresholds without first recording a baseline.
- Do not combine unrelated application refactors into tooling setup.

## Acceptance Notes

- F-009A provides a documented local quality command for tests, linting,
  formatting, dependency auditing, coverage policy, and module-growth checks.
- F-009B makes mandatory AI SDLC evidence, stage-specific review checklists,
  Definition of Done gates, and final-diff review mechanically verifiable.
- F-009C runs the same required local checks in continuous integration.
- F-009D adds browser workflow coverage for the current GM and player behavior
  before client refactoring and display features expand it.
- README, development documentation, Definition of Done, and agent instructions
  describe the resulting workflow consistently.

## Clarification Notes

- GitHub Actions is the planned F-009C continuous-integration provider because
  the repository is hosted on GitHub.
- F-009C should validate Node.js 22.8.0 or later in the Node 22 line, the
  documented minimum, and Node.js 24, the current development runtime.
- Use ESLint flat configuration for JavaScript correctness and Prettier for JS,
  JSON, CSS, HTML, and YAML formatting. Exclude Markdown initially to avoid an
  unrelated repository-wide documentation rewrite.
- High and critical dependency findings block completion. Moderate findings are
  reported but do not initially block unless reviewed and raised explicitly.
- Dependency-audit network failure is an explicit quality failure rather than a
  silent pass.
- Use Node's built-in test coverage, record the current baseline, and ratchet
  against regression rather than choosing an arbitrary initial target.
- Select the smallest browser-test approach that works with plain HTML and
  JavaScript in F-009D.
- Use a baseline ratchet rather than arbitrary file-size or coverage targets.
- Keep local and CI validation behavior identical.
- The approved implementation sequence completed E-005 and F-009A through
  F-009C; F-009D uses Playwright with Chromium as its initial automated target.
- The user explicitly approved the mandatory subagent workflow on 2026-06-20.
