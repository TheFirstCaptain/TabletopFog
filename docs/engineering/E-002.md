# E-002: Separate Campaign Storage Responsibilities

## Summary

Incrementally separate campaign schema normalization, validation,
serialization, and filesystem persistence into focused modules.

## Status

Done.

## Behavior to Preserve

Campaign creation, discovery, map management, ordering, persistence, and asset
containment continue to behave as documented and tested.

## Motivation

`server/campaign-storage.js` currently owns several distinct responsibilities.
Upcoming fog persistence will increase the cost and risk of changing it unless
the boundaries are made explicit first.

This work is queued before F-007 so fog persistence can build on clearer
storage boundaries instead of expanding the current combined module.

## Scope

- In scope: behavior-preserving extraction with existing public storage methods
  retained where practical.
- Out of scope: schema changes, new persistence behavior, fog storage, or new UI.

## Behavior Preservation Notes

- Public exports from `server/campaign-storage.js` remain available for callers
  and existing tests: `createCampaignStorage`, `createUserError`,
  `getDefaultDataRoot`, `normalizeFileName`, and `normalizePathSegment`.
- Campaign reads remain non-mutating and preserve unknown campaign and map
  metadata through later explicit writes.
- Rejected campaign, map, upload, metadata, order, active-map, and asset-path
  operations continue to leave files and metadata unchanged.
- Fog storage behavior is unchanged: in-memory fog remains outside
  `campaign.json` until F-007.
