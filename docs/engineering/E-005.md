# E-005: Establish Module Responsibility and Growth Baselines

## Summary

Record current source-module responsibilities and file-size baselines so future
quality checks can prevent further concentration without imposing arbitrary
retroactive limits.

## Status

Done.

## Behavior to Preserve

All application routes, storage behavior, GM/player workflows, commands, and
tests remain unchanged.

## Motivation

The repository asks agents to prefer small modules but has no measurable
baseline or escalation rule. After one-time Prettier normalization, current
concentration includes 429
lines in `server/campaign-storage.js`, 296 lines in `public/gm.js`, and 280 lines
in `server/index.js`. E-002 through E-004 own their later decomposition; this
record only establishes responsibilities, exceptions, and a no-regression
baseline for F-009A.

## Scope

- In scope: inventory production JavaScript modules, record primary
  responsibilities, establish current exceptions, and define how baselines are
  updated when modules shrink or move.
- Out of scope: refactoring modules, changing behavior, choosing arbitrary
  universal size limits, or implementing the automated check.

## Affected Systems

- GM view: responsibility baseline for `public/gm.js`.
- Local server: responsibility baselines for server modules.
- Tests and tooling: baseline data consumed by F-009A.
- Documentation: engineering tracker and quality-gate planning.

## Implementation Notes

- Treat existing oversized or multi-responsibility modules as temporary named
  exceptions rather than normal examples for new code.
- Use a ratchet: a module should not grow beyond its recorded baseline without a
  documented architecture disposition.
- Lower baselines after E-002, E-003, or E-004 extracts responsibilities.
- A new module requires an intentional baseline entry and non-empty
  responsibility. An architecture disposition is additionally required only
  when recording a temporary multi-responsibility exception; no arbitrary size
  ceiling is introduced.
- The approved machine-readable baseline should cover every current JavaScript
  file under `server/`, `public/`, and `scripts/`, require a non-empty
  responsibility description, reject unrecorded modules, and fail when a module
  grows beyond its recorded line count.
- Independent architecture review confirmed temporary responsibility exceptions
  for `server/campaign-storage.js`, `public/gm.js`, and `server/index.js`, linked
  to E-002, E-003, and E-004 respectively.
- Baseline mismatch in either direction should fail: growth requires an
  architecture disposition, while shrinkage requires lowering the stored value.
- Prettier normalization must occur before final line counts are frozen.

## Validation Plan

Confirm every production JavaScript file is represented, recorded line counts
match the repository, responsibility descriptions match actual imports and
exports, and the baseline format can be consumed by F-009A.

## Implementation Evidence

- Added `quality/module-baseline.json` with all 11 JavaScript modules under the
  three approved roots, exact physical line counts, responsibilities, and named
  temporary exceptions.
- Added a checker that rejects unrecorded, missing, grown, reduced, or
  responsibility-free modules, exact-root bypasses, and malformed temporary
  exceptions.
- Added red/green tests for exact inventory, growth, shrinkage, incomplete
  inventory, missing responsibilities, required-root bypasses, and malformed
  metadata.
- `npm run modules:check` passes with 11 recorded modules.
- Final clean-install `npm run quality` passes after independent code, test, and
  documentation reviews.
- E-002, E-003, and E-004 remain responsible for reducing the three temporary
  multi-responsibility exceptions.
