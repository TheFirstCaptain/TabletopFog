# E-011: Batch Brush Fog Operations

## Summary

Reduce Brush fog overhead by committing one brush stroke as a batch of fog
operations instead of sending one request, storage write, state broadcast, and
GM rerender for every sampled stamp.

## Status

Done.

## Behavior to Preserve

- Brush still paints circular hide/reveal stamps using the current spacing,
  diameter, and map-relative geometry.
- A Brush stroke remains one undoable user action.
- Player Display updates live only when the edited encounter is currently
  `Shown to Players`.
- Rejected operations leave storage, in-memory state, and undo history
  unchanged.
- Rectangle, Circle, Clear Fog, and Undo behavior remain unchanged.

## Motivation

E-010 found that `commitWorkspaceFogBrush()` loops over sampled brush points and
awaits `api.appendFogOperation()` for each one. Each stamp currently triggers a
campaign JSON write, state-store update, Socket.IO broadcast, full campaign
response, and GM rerender. Long brush strokes therefore scale with stamp count
rather than with the user's single stroke.

## Scope

- In scope:
  - Add a server/API path for appending multiple normalized fog operations to
    one encounter atomically.
  - Preserve one undo-history entry for the whole Brush stroke.
  - Update the GM Brush commit path to use the batch operation.
  - Add focused Node and Chromium regression coverage.
- Out of scope:
  - Changing Brush sampling, shape semantics, or UI labels.
  - Changing storage format.
  - Batching Rectangle or Circle tools.
  - Changing Player Display role boundaries.

## Affected Systems

- GM view: Brush commit path in `public/gm-controller.js` and API wrapper in
  `public/gm-api.js`.
- Player view: receives the same final fog state after one broadcast.
- Local server: fog operation route, state-store undo handling, and projection
  broadcast.
- Storage: same ordered fog operation list, written once per Brush stroke.
- Tests and tooling: focused server/state tests plus Chromium Brush workflow.
- Documentation: this engineering record and tracker.

## Implementation Notes

Prefer an additive batch endpoint or payload shape that keeps the existing
single-operation route compatible. Validate every operation before mutating
storage or memory. Apply storage and state updates in one sequence so partial
failure cannot leave storage, projection, or undo history divergent.

Implemented on 2026-07-09:

- Added `stateStore.appendFogOperations()` so a batch of normalized operations
  appends to one encounter with one undo-history entry.
- Added `POST /api/campaigns/:campaignId/maps/:mapId/fog-operations/batch` as
  an additive GM-only endpoint. The existing single-operation endpoint remains
  unchanged for Rectangle and Circle tools.
- Updated the GM Brush commit path to send one batch request per Brush stroke.
- Preserved the existing ordered fog storage shape; the batch endpoint writes
  the same `fog` operation list, just with fewer request/write/broadcast
  cycles.
- Rejected batches validate before storage or in-memory mutation, preserving
  campaign JSON, projected state, and undo history.
