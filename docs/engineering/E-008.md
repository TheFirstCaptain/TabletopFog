# E-008: Evaluate Local Datastore Adapter After Fog Persistence

## Summary

Evaluate whether TabletopFog should continue with local JSON campaign files or
introduce a local datastore adapter, likely SQLite, after F-007 persistence work
has exposed the real complexity of saving and restoring fog state.

## Status

Proposed.

## Behavior to Preserve

TabletopFog remains a local-first, single-GM application with inspectable,
portable campaign data. The GM can create/open campaigns, manage encounters,
show one encounter to players, and use fog without cloud hosting, accounts, or
a server database.

## Motivation

The current JSON file approach fits the MVP because it is simple, local,
portable, and easy to inspect. A datastore may become valuable only if F-007
shows concrete pressure around transactions, migrations, large fog operation
lists, indexing, crash recovery, or storage module complexity.

This record preserves the question without prematurely changing the storage
strategy before fog persistence exists.

## Scope

- In scope:
  - Review F-007 implementation complexity and defects after it is complete.
  - Identify concrete pain points in JSON persistence, if any.
  - Compare continued versioned JSON files against a local datastore adapter.
  - Consider SQLite as the likely local datastore candidate.
  - Recommend whether to stay with JSON, add a storage abstraction, prototype a
    SQLite adapter, or defer the decision.
- Out of scope:
  - Replacing campaign storage before F-007.
  - Adding a hosted database, cloud sync, accounts, or authentication.
  - Changing user-visible campaign workflows.
  - Migrating existing storage without a separate approved feature or decision.

## Affected Systems

- GM view: no direct behavior change expected.
- Player view: no direct behavior change expected.
- Local server: possible future storage adapter boundary.
- Storage: JSON files, schema versioning, atomic writes, future migrations, and
  possible SQLite adapter evaluation.
- Tests and tooling: storage contract tests, migration tests, crash/partial-write
  tests, and quality gate impact.
- Documentation: architecture, roadmap, development setup, and decision records
  if a datastore change is recommended.

## Implementation Notes

Do not start with a rewrite. First review the completed F-007 storage surface:
module boundaries, migration pressure, test friction, data volume, failure
modes, and recovery ergonomics.

If a datastore is recommended, prefer an adapter-style boundary so product code
does not depend directly on SQLite. Any approved migration should preserve a
clear backup/export story for local campaign data.

## Validation Plan

This is an evaluation item. Validation should include a written recommendation
with evidence from F-007, storage tests that characterize current behavior, and
an explicit decision on whether a follow-up feature, engineering item, or
decision record is needed.
