# E-008: Evaluate Local Datastore Adapter After Fog Persistence

## Summary

Evaluate whether TabletopFog should continue with local JSON campaign files or
introduce a local datastore adapter, likely SQLite, after F-007 persistence work
has exposed the real complexity of saving and restoring fog state.

## Status

Done.

## Behavior to Preserve

TabletopFog remains a local-first, single-GM application with inspectable,
portable campaign data. The GM can create/open campaigns, manage encounters,
show one encounter to players, and use fog without cloud hosting, accounts, or
a server database.

## Motivation

The current JSON file approach fits the MVP because it is simple, local,
portable, and easy to inspect. A datastore may become valuable only if F-007
shows concrete pressure around transactions, migrations, large fog operation
lists, indexing, crash recovery, or storage module complexity.

This record preserves the question without prematurely changing the storage
strategy before fog persistence exists.

## Scope

- In scope:
  - Review F-007 implementation complexity and defects after it is complete.
  - Identify concrete pain points in JSON persistence, if any.
  - Compare continued versioned JSON files against a local datastore adapter.
  - Consider SQLite as the likely local datastore candidate.
  - Recommend whether to stay with JSON, add a storage abstraction, prototype a
    SQLite adapter, or defer the decision.
- Out of scope:
  - Replacing campaign storage before F-007.
  - Adding a hosted database, cloud sync, accounts, or authentication.
  - Changing user-visible campaign workflows.
  - Migrating existing storage without a separate approved feature or decision.

## Affected Systems

- GM view: no direct behavior change expected.
- Player view: no direct behavior change expected.
- Local server: possible future storage adapter boundary.
- Storage: JSON files, schema versioning, atomic writes, future migrations, and
  possible SQLite adapter evaluation.
- Tests and tooling: storage contract tests, migration tests, crash/partial-write
  tests, and quality gate impact.
- Documentation: architecture, roadmap, development setup, and decision records
  if a datastore change is recommended.

## Implementation Notes

Do not start with a rewrite. First review the completed F-007 storage surface:
module boundaries, migration pressure, test friction, data volume, failure
modes, and recovery ergonomics.

If a datastore is recommended, prefer an adapter-style boundary so product code
does not depend directly on SQLite. Any approved migration should preserve a
clear backup/export story for local campaign data.

## Evaluation

Completed: 2026-07-04.

Recommendation: keep the current versioned JSON campaign storage for the near
term. Do not introduce SQLite, a datastore adapter, or a migration layer before
F-008.

Evidence from F-007:

- F-007A, F-007B, and F-007C completed fog persistence, shown-to-players
  restore, malformed fog recovery, missing asset safety, and storage-format
  documentation without requiring cross-campaign queries, indexing, concurrent
  writes, or background migrations.
- Storage complexity increased, but the pressure stayed inside focused modules:
  `server/campaign-schema.js` owns normalization and serialization,
  `server/campaign-files.js` owns path containment and atomic writes, and
  `server/campaign-storage.js` remains the public storage facade.
- Existing JSON write behavior is covered by tests for atomic metadata
  preservation, rejected mutation rollback, unknown metadata preservation,
  malformed fog recovery, missing asset recovery, and Player Display safety.
- The expected MVP data volume is still small: one local GM, campaign folders,
  map assets, and ordered rectangle fog operations. There is no current need for
  relational queries, multi-user locking, or large indexed lookups.
- SQLite would reduce some future transaction and migration concerns, but it
  would add a binary dependency, backup/export decisions, migration mechanics,
  and a less inspectable default data shape before there is a demonstrated
  product need.

Known JSON costs to watch:

- `campaign.json` is rewritten on each successful campaign mutation. This is
  acceptable for current map and rectangle-fog payload sizes, but it may become
  pressure if future brush tools create very large operation histories.
- Recovery and forward-compatibility logic must remain carefully tested because
  reads normalize into runtime state while writes serialize the supported shape.
- If future work adds undo stacks, large freehand fog, notes, session logs, or
  multiple independent local clients, revisit storage pressure before
  implementation.

Decision: defer datastore work. If concrete pressure appears later, create a new
engineering item or decision record for one of these narrower options:

- Add a storage contract test suite around the existing JSON facade.
- Extract an adapter interface while keeping the JSON adapter as the only
  implementation.
- Prototype SQLite in a separate branch or spike with explicit backup/export
  and migration requirements.
