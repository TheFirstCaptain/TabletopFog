# E-003: Separate GM Client Responsibilities

## Summary

Separate GM API access, client state coordination, DOM rendering, and event
wiring into small plain-JavaScript modules.

## Status

Done.

## Behavior to Preserve

The campaign library and map-management workflows remain unchanged, including
current labels, messages, and Socket.IO synchronization.

## Motivation

`public/gm.js` currently combines transport, mutable state, rendering, and event
handling in one module. Smaller boundaries will make future map and fog work
easier to test and review.

## Scope

- In scope: behavior-preserving extraction using browser-native modules.
- Out of scope: a frontend framework, build system, visual redesign, or new GM
  controls.

## Affected Systems

- GM view: internal module loading and responsibility boundaries only.
- Player view: unchanged; used to verify synchronization remains intact.
- Local server: unchanged routes and payloads; static module serving remains
  through the existing public directory.
- Storage: unchanged.
- Tests and tooling: Chromium characterization, served-module assertions,
  lint configuration, and module responsibility baseline.
- Documentation: engineering record and tracker; README remains unchanged
  because commands and user-visible workflows do not change.

## Implementation Notes

- `public/gm-api.js` owns fetch requests, endpoint construction, headers, and
  response error conversion.
- `public/gm-state.js` owns campaign-library and current-campaign state plus
  reorder and Socket.IO synchronization decisions.
- `public/gm-view.js` owns DOM references and rendering without API or socket
  access.
- `public/gm-events.js` owns static and delegated browser event wiring without
  transport or rendering decisions.
- `public/gm-controller.js` coordinates API results, state updates, view
  refreshes, messages, and socket callbacks.
- `public/gm.js` becomes the browser bootstrap that composes those modules.
- Load the GM entry point with `type="module"`; retain the global Socket.IO
  client supplied by the existing preceding script.
- Extract incrementally and do not rename visible text, change request shapes,
  or add test-only production hooks.
- Roll back the extraction as one unit if browser characterization exposes a
  behavior change that cannot be corrected within these boundaries.
