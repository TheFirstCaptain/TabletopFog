# E-003: Separate GM Client Responsibilities

## Summary

Separate GM API access, client state coordination, DOM rendering, and event
wiring into small plain-JavaScript modules.

## Status

Proposed.

## Behavior to Preserve

The campaign library and map-management workflows remain unchanged, including
current labels, messages, and Socket.IO synchronization.

## Motivation

`public/gm.js` currently combines transport, mutable state, rendering, and event
handling in one module. Smaller boundaries will make future map and fog work
easier to test and review.

## Scope

- In scope: behavior-preserving extraction using browser-native modules.
- Out of scope: a frontend framework, build system, visual redesign, or new GM
  controls.

## Validation Plan

Run automated tests and exercise campaign creation, open, upload, rename,
reorder, active-map selection, error messages, and reconnect synchronization.
