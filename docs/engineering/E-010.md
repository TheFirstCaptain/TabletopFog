# E-010: Performance Review and Improvement Plan

## Summary

Review TabletopFog performance across the current GM View, Player Display,
local server, storage, and test/tooling paths, then identify concrete follow-up
work where measured evidence shows improvement is worthwhile.

This is an audit and planning maintenance item first. It should not bundle
performance fixes unless a change is tiny, behavior-preserving, and clearly
validated inside the review.

## Status

Done.

## Behavior to Preserve

- Opening or editing an encounter remains GM-local and never changes the Player
  Display.
- Only explicit `Show to Players` and clear actions change the Player Display.
- Player Display remains read-only.
- Campaign reads remain non-destructive and storage stays inspectable local
  JSON plus map assets.
- Fog rendering, undo, Clear Fog, grid alignment, zoom, pan, and fullscreen
  behavior remain visually and semantically unchanged.
- Existing documented commands, HTTPS behavior, and quality gates remain
  unchanged unless a separately reviewed follow-up requires updates.

## Motivation

The app now has enough V1 surface area to make performance worth reviewing
before future polish accumulates: campaign library rendering, encounter
gallery rendering, canvas map/fog drawing, Socket.IO updates, JSON storage, map
asset serving, and Chromium workflow tests are all active. A focused review can
separate measured bottlenecks from guesses and create small follow-up records
for improvements that matter at the table.

Expected benefit:

- Establish a repeatable performance baseline for current workflows.
- Identify avoidable latency, rendering work, memory growth, or test slowness.
- Prioritize improvements by user impact and risk instead of code aesthetics.
- Reclassify any broken promised behavior as bugs instead of hiding it inside
  maintenance.

## Scope

- In scope:
  - Measure representative GM and Player Display workflows in browser tests or
    local profiling where practical.
  - Review canvas map/fog render paths for unnecessary redraws, allocations,
    image handling, or geometry recomputation.
  - Review GM View rendering paths for campaign and encounter lists, workspace
    updates, and control-state churn.
  - Review Socket.IO/API update flow for redundant broadcasts or payloads.
  - Review local JSON storage operations for unnecessary synchronous work on
    hot paths.
  - Review automated test runtime for obvious harness bottlenecks that do not
    weaken coverage.
  - Produce prioritized follow-up recommendations with evidence, expected
    benefit, and risk.
- Out of scope:
  - Product behavior changes.
  - Storage format migrations.
  - New dependencies, build systems, frontend frameworks, or databases.
  - Authentication, cloud hosting, or remote telemetry.
  - Broad rewrites based only on suspicion.
  - Changes that weaken Player Display read-only boundaries or recovery
    guarantees.

## Affected Systems

- GM view: campaign library, campaign page, Encounter Workspace, controls, and
  map/fog canvas interactions may be measured and reviewed.
- Player view: read-only render, fullscreen, local zoom/pan, reconnect, and
  image-error flows may be measured and reviewed.
- Local server: API routes, Socket.IO projection, image serving, and in-memory
  state updates may be measured and reviewed.
- Storage: campaign JSON reads/writes and map asset metadata handling may be
  measured and reviewed without changing format.
- Tests and tooling: Node and Chromium test runtime may be profiled for
  obvious, behavior-preserving speedups.
- Documentation: this record, the engineering tracker, and any follow-up bug or
  engineering records created from findings.

## Implementation Notes

Start with measurement and code-path review before editing production code.
Use small representative scenarios such as:

- Empty campaign library, many campaign cards, and malformed campaign
  diagnostics.
- Campaign page with multiple encounters and varied thumbnail sizes.
- Encounter Workspace drawing rectangle, Brush, Circle, undo, Clear Fog, grid,
  zoom, and pan.
- Player Display receiving shown encounter changes, fog updates, reconnects,
  and image-load failures.
- Campaign open with persisted fog and recovery diagnostics.

Any resulting implementation work should be split into follow-up records when
it changes shared render paths, server projection, storage behavior, validation
commands, or test harness behavior. If review finds a user-visible slowdown
that violates promised behavior, create or link a bug.

## Review Evidence

Completed on 2026-07-09.

Baseline measurements:

- `npm run test:coverage`: 121 Node tests passed in about 1.3 seconds of
  reported test runtime.
- `npm run test:browser`: 31 Chromium tests passed in about 21.1 seconds with
  the configured single worker.
- `npm run test:browser -- --workers=4`: 31 Chromium tests passed in about
  19.5 seconds; Playwright still used only 2 workers because tests were not
  fully parallel.
- `npm run test:browser -- --fully-parallel --workers=4`: 31 Chromium tests
  passed in about 7.7 seconds using 4 workers.

Code-path review:

- `public/gm-controller.js` sends Brush fog stamps one at a time. Each stamp
  calls `api.appendFogOperation`, causing a server read/write, undo-history
  update, Socket.IO sync, full GM state response, and `renderCurrentCampaign()`.
- `server/http-routes.js` persists each appended fog operation by reading the
  current map state, writing the full campaign JSON, updating in-memory state,
  and broadcasting the new state.
- `public/map-canvas.js` rebuilds a full-stage mask canvas and replays every
  fog operation on every draw, including pan, zoom, and resize draws where the
  fog operation list has not changed.
- `public/gm-view.js` rebuilds the full encounter gallery on every
  `renderCampaign()` call. Fog-only changes in the selected Encounter Workspace
  still recreate encounter cards and thumbnail image elements.
- `server/socket-sync.js` broadcasts full role projections on every state
  change. This remains acceptable at current V1 scale, but it compounds the
  Brush stamp issue because one drag can produce many state changes.
- `browser-test/fixtures.js` already isolates each Chromium test with a
  temporary data root, generated certificate, random local port, and fresh
  browser context, which makes parallel browser execution plausible.

## Findings

| ID | Area | Severity | Evidence | Recommendation | Follow-Up |
| --- | --- | --- | --- | --- | --- |
| E-010-F001 | Brush fog persistence | Medium | Brush commits loop through every sampled stamp and make one HTTP request, campaign JSON write, state broadcast, and GM rerender per stamp. | Add a batch append path so one Brush stroke is persisted, undoable, broadcast, and rendered as one logical action. | [E-011](./E-011.md) |
| E-010-F002 | Canvas fog rendering | Medium | `drawFog()` resizes the mask canvas and replays all fog operations on every draw, so pan/zoom cost grows with accumulated fog operations. | Cache or otherwise reduce repeated fog-mask work while preserving ordered hide/reveal semantics and role-specific opacity. | [E-012](./E-012.md) |
| E-010-F003 | GM workspace rendering | Low | `renderCampaign()` calls `renderMaps()` for fog-only workspace changes, recreating encounter cards and thumbnail images even when gallery state is unchanged. | Split selected-workspace rendering from gallery rendering or add a narrow update path for fog-only state changes. | [E-013](./E-013.md) |
| E-010-F004 | Browser workflow runtime | Low | Fully parallel Chromium execution passed locally and reduced runtime from about 21.1 seconds to about 7.7 seconds. | Review and, if stable in CI, enable parallel browser workflows without weakening isolation or diagnostics. | [E-014](./E-014.md) |

No broken promised behavior was found during this review, so no bug record was
created.
