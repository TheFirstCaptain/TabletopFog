# E-001: Clean Up Test Temporary Directories

## Summary

Ensure automated tests remove temporary campaign data and certificate files
after each run.

## Status

Done.

## Behavior to Preserve

All existing automated tests continue to run independently and validate the same
application behavior.

## Motivation

Tests create directories under the operating system temporary root but do not
remove them. Repeated runs leave avoidable filesystem artifacts.

## Scope

- In scope: shared temporary-resource registration and reliable teardown.
- Out of scope: changing application storage behavior or certificate generation.

## Implementation Plan

- Add one test-support helper that creates an operating-system temporary
  directory and registers recursive, idempotent removal with the current test
  context.
- Use the helper for campaign storage roots, server data roots, generated test
  certificates, and certificate-regeneration tests.
- Keep server/socket shutdown in existing `finally` blocks; directory teardown
  complements rather than replaces resource closure.
- Add focused coverage proving the registered teardown removes nested files and
  tolerates an already-removed directory.

## Affected Systems

- Tests and tooling: shared test support plus campaign storage, certificate,
  server, harness, and module-baseline tests.

## Validation Plan

Run the focused helper, campaign, certificate, and server tests, then
`npm run quality`. Confirm the helper's registered cleanup removes nested test
artifacts and remains safe when cleanup follows a failure or prior removal.

## Implementation Evidence

- Added `test-support/temp-directory.js` as the only direct creator of
  operating-system temporary test directories.
- Migrated campaign, certificate, server, harness, and module-baseline tests to
  register teardown through the current Node test context.
- Added focused coverage proving recursive cleanup removes nested artifacts and
  remains safe when invoked after the directory is already absent.
- Existing explicit server and Socket.IO closure remains in `finally` blocks.
- Focused cleanup and HTTPS server suites passed; the authoritative
  `npm run quality` run passed 68 tests.
- No production module, storage behavior, protocol, or user workflow changed.
- README and acceptance tests required no update because commands and
  user-visible behavior are unchanged; no bug or feature reclassification was
  needed.
