# E-012: Optimize Canvas Fog Mask Rendering

## Summary

Reduce repeated canvas fog-mask work when panning, zooming, resizing, or
redrawing maps with accumulated fog operations.

## Status

Done.

## Behavior to Preserve

- Ordered hide/reveal replay remains visually identical.
- Rectangle, Brush, and Circle fog geometry remains map-relative.
- GM workspace fog remains semi-transparent and Player Display fog remains
  opaque or near-opaque.
- Zoom, pan, grid alignment, fullscreen, image-error, and reconnect behavior
  remain unchanged.

## Motivation

E-010 found that `public/map-canvas.js` rebuilds the full-stage mask canvas and
replays every fog operation on every draw. That is simple and correct, but pan
and zoom redraws become more expensive as fog operations accumulate even when
the operation list itself has not changed.

## Scope

- In scope:
  - Measure fog draw cost with representative operation counts.
  - Explore caching or reducing repeated mask work without changing visual
    output.
  - Preserve current canvas renderer API where practical.
  - Add regression coverage for role opacity and ordered reveal/hide behavior.
- Out of scope:
  - Raster storage format changes.
  - Replacing Canvas with SVG, WebGL, or a frontend framework.
  - Changing fog semantics or user controls.

## Affected Systems

- GM view: Encounter Workspace canvas rendering.
- Player view: Player Display canvas rendering.
- Local server: not expected to change.
- Storage: not expected to change.
- Tests and tooling: renderer-focused tests and existing Chromium workflows.
- Documentation: this engineering record and tracker.

## Implementation Notes

Candidate approaches include caching normalized operation paths, caching a
map-relative mask that is transformed into the stage, or short-circuiting
expensive work when the operation list and draw metrics have not changed. Any
approach must account for ordered `destination-out` reveal behavior and role
opacity differences.

Implemented on 2026-07-09:

- Changed the renderer to build the fog mask in map/image coordinates rather
  than stage coordinates.
- The renderer now reuses the same ordered fog mask across pan, zoom, resize,
  and ordinary redraws, drawing it into the current map rectangle with the same
  role-specific opacity.
- The mask is marked dirty only when the map/image source changes or the
  normalized fog operation list changes.
- Added `data-fog-mask-builds` as browser-test diagnostic state so workflow
  coverage can prove zoom and resize do not rebuild the mask.

## Validation Plan

- Add or extend renderer tests for unchanged visual semantics with mixed hide
  and reveal operations.
- Run focused Chromium fog workflows.
- Run `npm run quality`.
