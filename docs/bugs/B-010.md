# B-010: Locked Grid Does Not Scale With GM Map Zoom

## Summary

The GM workspace grid stays fixed in screen space after locking instead of
scaling with the map, and the zoom percentage readout wraps awkwardly between
the zoom buttons.

## Status

Resolved.

## Observed Behavior

After the GM locks the grid and uses the GM workspace map zoom controls, the
map changes size but the locked grid remains the same screen-space size. The
zoom readout can also wrap between `Zoom out` and `Zoom in` in the compact Map
Alignment panel.

## Expected Behavior

Before locking, the grid acts as a screen-space alignment aid. After locking,
the grid is anchored to the map and scales with GM map zoom. The zoom readout
sits below the zoom buttons in a compact stack.

## Reproduction

1. Open a campaign with an encounter map.
2. Open the Encounter Workspace.
3. Show the 5 ft grid.
4. Lock the grid.
5. Use the GM `Zoom in` or `Zoom out` controls.
6. Observe the grid and the Map Alignment zoom control layout.

## Affected Systems

- GM view: Map Alignment controls and locked grid overlay.
- Player view: no expected effect.
- Local server: not affected.
- Map/fog rendering: GM-only grid overlay and GM-local map zoom interaction.
- Storage: not affected.
- Network/device setup: not affected.
- Documentation: bug tracker and related F-006A evidence.

## Investigation Notes

Reported from manual GM View inspection on 2026-07-01 with an attached crop of
the Map Alignment panel showing the zoom readout wrapping between the zoom
buttons.

The fix makes the GM zoom controls a compact grid with `Zoom out` and `Zoom in`
on the first row, the percentage readout centered below them, and `Fit map`
below the readout. Browser regression coverage now checks both states: unlocked
grid metrics remain screen-space while the map zooms underneath, and locked
grid cell size scales with the map while the overlay remains clipped to the map
stage.
