# B-006: Harness Repository Test Rejects Current-Dated Evidence

## Summary

The live-repository harness test uses a fixed June 20 clock and rejects valid
feature evidence recorded on later dates as future-dated.

## Status

Resolved.

## Observed Behavior

On June 21, `npm test` reports every June 21 F-009D SDLC stage as having an
invalid future completion date even though the standalone harness command uses
the real current date and passes.

## Expected Behavior

Deterministic policy unit tests may use a fixed clock, but the test validating
the current repository must evaluate evidence against the actual current date.

## Reproduction

1. Record valid Harness Version 1 evidence dated after 2026-06-20.
2. Run `npm test`.
3. Observe `current repository conforms to the harness contract` fail because
   it passes a fixed `2026-06-20` clock to repository validation.

F-009D planning evidence reproduced this on 2026-06-21. The standalone
`npm run harness:check` passed against the same files.

## Affected Systems

- Tests and tooling: harness compliance repository integration test.
- Documentation: any governed feature evidence recorded after 2026-06-20.

## Investigation Notes

The fixed clock remains appropriate for synthetic policy cases. Only the test
that loads the live repository should use the validator's current-date default.

## Validation Plan

Change the live-repository assertion to use the default current clock, retain
fixed time for synthetic policy tests, rerun the focused harness tests, and run
the authoritative quality command.

## Resolution

The live-repository test now uses the validator's current-clock default. Fixed
clocks remain unchanged for synthetic policy cases.

## Validation Evidence

- The initial complete Node run rejected five valid June 21 F-009D stages as
  future-dated.
- Focused harness tests passed 22 tests after the correction.
- `npm run quality` passed 75 Node tests, 4 Chromium workflows, coverage floors,
  policy checks, and a zero-vulnerability high-severity audit.
