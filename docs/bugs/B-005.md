# B-005: Hidden Player Image Remains Visible After Load Failure

## Summary

The player reports an active-map image load failure but Chromium continues to
render the broken image element instead of hiding it.

## Status

Resolved.

## Observed Behavior

`player.js` sets the map image's `hidden` property after an image error, but
`.player-map { display: block; }` overrides the browser's hidden presentation.
The error message appears while the broken image element remains visible.

## Expected Behavior

After an active-map image fails to load, the player displays `Map image could
not be loaded.` and hides the failed image element.

## Reproduction

1. Open GM and player pages in Chromium.
2. Add a valid map.
3. Make `/api/player/active-map/asset` return an error.
4. Select the map for players.
5. Observe the error message and the still-visible broken image element.

F-009D browser characterization reproduced this on 2026-06-21. Its initial
image-failure test failed because the image remained visible despite having the
`hidden` attribute.

## Affected Systems

- Player view: failed active-map presentation.
- Map/fog rendering: static active-map image visibility.
- Documentation: bug tracker and F-009D findings.

## Investigation Notes

Both `.player-map` and `.map-preview` declare `display: block`. A global
`[hidden]` rule is the narrow semantic correction because several existing GM
and player elements rely on the standard hidden attribute.

## Validation Plan

Keep the failing Chromium image-error regression test, add the smallest CSS
rule that makes hidden elements non-rendered, rerun the browser suite, and run
the authoritative quality command. Review the change for GM panel and preview
visibility regressions.

## Resolution

Added a global `[hidden]` presentation rule so explicit hidden state wins over
the existing GM preview and player map display declarations.
