# B-002: Campaign Reads Can Destructively Normalize Metadata

## Summary

Listing or opening campaigns can rewrite `campaign.json` and discard fields not
recognized by the current serializer.

## Status

Open.

## Observed Behavior

Campaign storage reads frequently pass `save: true`. Normalized data is then
serialized using only known fields, so a read operation can change persisted
data and remove forward-compatible or externally added metadata.

## Expected Behavior

Read operations should not modify campaign files. Explicit migrations should be
validated, versioned, and preserve data according to a documented policy.

## Reproduction

1. Add an unknown metadata field to a valid `campaign.json`.
2. List or open the campaign through campaign storage.
3. Read `campaign.json` again.
4. Observe that normalization rewrote the file and removed the unknown field.

## Affected Systems

- Local server: campaign list and open API behavior.
- Storage: `campaign.json` integrity and future migration compatibility.

## Investigation Notes

Current order-repair tests intentionally expect repair-on-read behavior. The fix
must define when repair or migration writes are allowed rather than simply
removing test coverage.

## Validation Plan

Add a failing regression test for non-mutating reads and unknown-field handling,
implement an explicit migration/write boundary, and run the full storage and
server test suites.
